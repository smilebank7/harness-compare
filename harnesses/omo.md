## 개요

omo는 opencode의 플러그인 API 위에 올라가는 "배터리 포함" 하네스다. 자기 소개부터 "멀티 모델 오케스트레이션, 병렬 백그라운드 에이전트, 수제 LSP/AST 도구를 갖춘 opencode 플러그인"이고([package.json#L4](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/package.json#L4)), 같은 제품을 두 에디션으로 배포한다: opencode용 **Ultimate**(11 에이전트, 54+ 라이프사이클 훅, 5종 내장 MCP, Team Mode, ulw-loop, ultrawork, hashline 편집 전부)와 Codex CLI용 **Light**(에이전트 오케스트레이션을 뺀 이식 가능 컴포넌트만)다([README.md#L114-L117](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/README.md#L114-L117)). 레포는 현재 OpenCode·Codex·Pi 등 복수 하네스를 지원하는 "Multi-Harness Agent OS"로의 개편이 진행 중이라고 명시한다([README.md#L11-L14](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/README.md#L11-L14)).

제작 의도의 핵심은 기준선과의 대비로 읽힌다. opencode가 "어느 모델에도 묶이지 않는 중립 하네스"라면, omo는 그 위에 **의견이 강하게 들어간 운영 체제**를 얹는다: 작업 종류별로 모델을 미리 골라 두고(카테고리 라우팅), 에이전트마다 모델 폴백 체인을 박아 두고(AGENT_MODEL_REQUIREMENTS), 모델이 멈추면 다시 굴리는 장치(todo-continuation, ralph-loop)를 둔다. 플러그인 한 장이지만 사실상 베이스의 도구 표면·프롬프트·모델 선택을 전부 덮어쓰는 재구성이다.

진입 형태는 기준선이 정의한 플러그인 계약([opencode.md 아키텍처](opencode.md#아키텍처)) 그대로다: 단일 `PluginInterface` 객체가 `tool`, `chat.params`, `chat.message`, `experimental.chat.system.transform`, `config`, `event`, `tool.definition`, `tool.execute.before/after` 훅을 한꺼번에 반환한다([plugin-interface.ts#L36-L103](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/plugin-interface.ts#L36-L103)).

## 하네스 엔지니어링

framework.md 정의: [하네스 엔지니어링](../framework.md#1-하네스-엔지니어링-harness-engineering).

**도구 표면의 확장.** omo는 베이스 도구 위에 자체 도구군을 등록한다: 서브에이전트 직접 호출(`call_omo_agent`), 멀티모달 분석(`look_at`), 카테고리 위임(`delegate_task`), 태스크 CRUD(`task_*`), 라인 해시 편집(`hashline_edit`), 팀 메시징, 그리고 백그라운드 태스크 조회·취소(`background_output`/`background_cancel`)다([tools/index.ts#L24-L34](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/index.ts#L24-L34), [#L36-L43](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/index.ts#L36-L43)). tmux 기반 `interactive_bash`도 별도 표면이다([tools/index.ts#L9](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/index.ts#L9)). 레지스트리는 코어 도구 + 팀 모드 도구 + 태스크 시스템 도구 + 해시라인 도구를 조건부 결합하고, 비활성 도구 필터와 `experimental.max_tools` 상한 트리밍까지 자체 처리한다([tool-registry.ts#L51-L64](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/plugin/tool-registry.ts#L51-L64), [#L78-L83](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/plugin/tool-registry.ts#L78-L83)).

**라인 해시 앵커 편집(hashline).** 기준선 edit가 문자열 매칭 복구 파이프라인([opencode.md 하네스](opencode.md#하네스-엔지니어링))이라면, omo는 편집 좌표계 자체를 바꾼다. read 도구 출력의 각 라인을 `tool.execute.after` 훅에서 `LINE#HASH|content` 형식으로 후처리해 모델에게 보여주고([hashline-read-enhancer/hook.ts#L56-L66](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/hashline-read-enhancer/hook.ts#L56-L66)), 해시는 정규화된 라인 내용의 xxhash32를 256개 토큰 사전에 사상해 만든다([hash-computation.ts#L7-L13](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/hashline-core/src/hash-computation.ts#L7-L13), [#L15-L26](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/hashline-core/src/hash-computation.ts#L15-L26)). `hashline_edit` 도구는 이 앵커(`pos`/`end`, `LINE#ID` 형식)를 받아 replace/append/prepend 연산을 적용한다([hashline-edit/tools.ts#L21-L38](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/hashline-edit/tools.ts#L21-L38)). 모델이 옛 컨텍스트의 라인을 짚으면 해시 불일치로 거부되므로, 환각 편집 방어를 문자열 매칭이 아니라 좌표 검증으로 옮긴 설계다. framework.md의 경계 예시("edit 도구의 라인 해시 검증 = 하네스")에 정확히 해당한다.

**내장 MCP 주입.** 플러그인의 `config` 훅에서 opencode 설정의 `mcp` 맵을 직접 재작성해 websearch, context7, grep_app, lsp, ast_grep 5종 MCP를 주입한다([mcp/index.ts#L26-L57](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/mcp/index.ts#L26-L57)). 병합 우선순위는 내장 < Claude Code `.mcp.json` < 사용자 설정 < 플러그인 컴포넌트 순이고, 사용자가 비활성화한 항목은 끝까지 존중한다([mcp-config-handler.ts#L50-L68](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/plugin-handlers/mcp-config-handler.ts#L50-L68)). LSP MCP는 동봉된 `lsp-daemon`/`lsp-tools-mcp` 패키지를 찾고, 빌드 산출물이 없으면 npm install→build→실행까지 수행하는 부트스트랩 스크립트를 한 줄 명령으로 인라인한다([mcp/lsp.ts#L7-L11](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/mcp/lsp.ts#L7-L11), [#L12-L32](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/mcp/lsp.ts#L12-L32)) — LSP/AST를 "도구 표면의 일부"로 보장하려는 의지가 패키징까지 침투한 사례다.

**컨텍스트 공급 장치.** 파일을 읽거나 편집할 때마다 경로에 매칭되는 규칙 파일을 찾아 도구 출력에 덧붙이는 rules-injector가 있다. 실경로·내용 해시 이중 중복 제거와 매치 판정 캐시를 갖춘 본격적인 주입 파이프라인이다([rules-injector/injection-processor.ts#L46-L54](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/rules-injector/injection-processor.ts#L46-L54), [#L71-L78](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/rules-injector/injection-processor.ts#L71-L78)). 같은 결의 가드·공급 훅들(디렉토리 AGENTS.md/README 주입, 이미지 자동 리사이즈, 기존 파일 write 가드, bash로 파일 읽기 가드 등)이 54종 훅 enum에 이름으로 고정되어 있다([config/schema/hooks.ts#L3-L60](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/config/schema/hooks.ts#L3-L60)).

## 루프 엔지니어링

framework.md 정의: [루프 엔지니어링](../framework.md#2-루프-엔지니어링-loop-engineering).

**todo 강제 연속(todo-continuation-enforcer).** 기준선의 루프가 "종료가 기본"([opencode.md 루프](opencode.md#루프-엔지니어링))이라면 omo는 그 종료 판정을 불신한다. 세션이 idle로 떨어졌는데 미완료 todo가 남아 있으면 연속 프롬프트를 주입해 다시 굴리고, 그 프롬프트는 "전부 끝냈다고 믿는다면 시스템이 당신의 완료 주장을 의심하는 것이니 회의적 관점에서 각 todo를 재검증하라"고 명시한다([constants.ts#L7-L14](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/todo-continuation-enforcer/constants.ts#L7-L14)). 다만 무한 재가동을 막는 스킵 조건이 길게 늘어선다: 복구 중, 사용자 취소, 토큰 한도 오류, 직전 abort, 백그라운드 태스크 실행 중이면 개입하지 않으며([idle-event.ts#L42-L84](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/todo-continuation-enforcer/idle-event.ts#L42-L84)), 정체 감지 3회·연속 실패 5회 상한 상수가 박혀 있다([constants.ts#L20-L25](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/todo-continuation-enforcer/constants.ts#L20-L25)).

**ralph-loop(ulw-loop).** 세션 단위의 명시적 반복 루프 컨트롤러다. `startLoop(sessionID, prompt, {maxIterations, completionPromise, ultrawork, strategy: "reset"|"continue"})` / `resumeLoop` / `cancelLoop` 인터페이스로 같은 프롬프트를 완료 약속(completionPromise)이 충족될 때까지 재투입한다([ralph-loop-hook.ts#L10-L26](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/ralph-loop/ralph-loop-hook.ts#L10-L26)). 기준선의 루프가 "한 턴을 더 돌지"의 문제라면 ralph-loop은 "한 세션을 더 돌지"라는 한 단계 위의 반복 제어다.

**서브에이전트 수명 주기: 동기/비동기 이원화.** `delegate_task`는 `run_in_background` 플래그로 동기 대기와 비동기 백그라운드 실행을 가르고, `task_id`(세션 id)로 이전 위임 세션을 이어가는 continuation 경로를 제공한다([delegate-task/tools.ts#L45-L54](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/delegate-task/tools.ts#L45-L54), [#L96-L101](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/delegate-task/tools.ts#L96-L101)). 백그라운드 태스크는 세션 lineage를 따라 올라가며 깊이를 계산해 기본 3단까지만 재귀 스폰을 허용하고, 계보 확인이 실패하면 스폰 자체를 차단한다([subagent-spawn-limits.ts#L4](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/features/background-agent/subagent-spawn-limits.ts#L4), [#L16-L53](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/features/background-agent/subagent-spawn-limits.ts#L16-L53)). 불안정 모델용 에이전트는 카테고리 설정의 `is_unstable_agent` 플래그로 "감시를 위한 강제 백그라운드 모드"에 들어간다([categories.ts#L24-L25](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/config/schema/categories.ts#L24-L25)).

**복구 훅 군집.** 편집 실패 복구, JSON 오류 복구, 위임 실패 재시도 가이드, 중단된 연속 가드, 세션 복구, Anthropic 컨텍스트 한도 초과 복구, 선제 컴팩션 등이 전부 독립 훅으로 존재하며 이름이 스키마에 열거된다([hooks.ts#L5](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/config/schema/hooks.ts#L5), [#L34-L36](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/config/schema/hooks.ts#L34-L36), [#L15-L16](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/config/schema/hooks.ts#L15-L16), [#L47](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/config/schema/hooks.ts#L47)). 기준선이 재시도를 단일 정책(지수 백오프)으로 두는 것과 달리, omo는 실패 양상별 전용 복구 경로를 늘어놓는 접근이다.

## 프롬프트 엔지니어링

framework.md 정의: [프롬프트 엔지니어링](../framework.md#3-프롬프트-엔지니어링-prompt-engineering).

**ultrawork 키워드 훅.** 사용자 메시지에서 키워드(ultrawork/hyperplan 등)를 감지해 행동 지시문을 주입하는 장치가 `chat.message` 훅에 산다. 합성 메시지·시스템 지시·슬래시 커맨드·비-omo 에이전트·백그라운드 세션은 건너뛰고, 플래너 에이전트에서는 ultrawork류 키워드를 걸러낸다([keyword-detector/hook.ts#L96-L107](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/keyword-detector/hook.ts#L96-L107), [#L109-L115](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/keyword-detector/hook.ts#L109-L115)). `default_mode.ultrawork`를 켜면 키워드 없이도 시스템 프롬프트에 `<ultrawork-mode>` 블록을 상시 주입하되, 컴팩션 후 재주입 중복을 태그 검사로 막는다([system-transform.ts#L26-L36](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/plugin/system-transform.ts#L26-L36)).

**에이전트 로스터 = 페르소나 프롬프트 묶음.** 빌트인 에이전트 11종(sisyphus, hephaestus, prometheus, oracle, librarian, explore, multimodal-looker, metis, momus, atlas, sisyphus-junior)이 스키마에 고정되고([agent-names.ts#L3-L15](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/config/schema/agent-names.ts#L3-L15)), 각각 팩토리 함수로 페르소나 프롬프트를 생성한다([builtin-agents.ts#L32-L45](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/agents/builtin-agents.ts#L32-L45)). 주목할 점은 프롬프트가 정적 텍스트가 아니라 **런타임 합성물**이라는 것: 메인 에이전트(sisyphus)의 프롬프트는 현재 사용 가능한 도구·스킬·카테고리·동료 에이전트 메타데이터를 인자로 받아 위임 표와 트리거 절을 동적으로 생성한다([builtin-agents.ts#L47-L59](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/agents/builtin-agents.ts#L47-L59), [sisyphus-agent-factory.ts#L32-L43](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/agents/sisyphus-agent-factory.ts#L32-L43)). 기준선의 ".txt 자산" 접근([opencode.md 프롬프트](opencode.md#프롬프트-엔지니어링))과 대비되는 코드-합성 프롬프트다.

**호출자 대상 프롬프트.** 카테고리마다 위임 프롬프트에 자동 첨부되는 `prompt_append`가 있는데, 수신 에이전트가 아니라 **호출하는 쪽**을 겨냥한 절이 들어 있다: "이 카테고리는 중간 티어 모델을 쓰므로 MUST DO/MUST NOT DO/EXPECTED OUTPUT을 명시하라"는 Caller_Warning, "다른 카테고리에 맞으면 선택하지 말라"는 Selection_Gate가 그 예다([anthropic-categories.ts#L17-L24](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/delegate-task/anthropic-categories.ts#L17-L24), [#L6-L14](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/delegate-task/anthropic-categories.ts#L6-L14)). 위임 품질을 수신자 능력이 아니라 발신자 지시 품질로 끌어올리려는 프롬프트 설계다.

**완료 의심 프롬프트.** 루프 절에서 본 CONTINUATION_PROMPT의 내용 자체("시스템이 당신의 완료 주장을 의심한다")도 프롬프트 차원의 산출물이다([constants.ts#L7-L14](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/todo-continuation-enforcer/constants.ts#L7-L14)) — 주입 시점은 루프, 텍스트는 프롬프트로 갈리는 경계 사례다.

## 아키텍처

framework.md 정의: [아키텍처](../framework.md#4-아키텍처-architecture).

**플러그인 1장 + 하네스 중립 코어의 모노레포.** 산출물은 opencode 플러그인 하나지만, 레포는 `hashline-core`, `model-core`, `prompts-core`, `rules-engine`, `boulder-state`, `agents-md-core` 등 하네스 중립 코어 패키지와 어댑터 2종(`omo-opencode`, `omo-codex`)으로 분해된 워크스페이스다([package.json#L8-L23](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/package.json#L8-L23)). 이 경계는 관습이 아니라 테스트로 강제된다: prompts-core 소스에 `@opencode-ai` import가 하나라도 있으면 실패하는 결합도 감사 테스트가 있다([opencode-coupling-audit.test.ts#L9-L13](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/prompts-core/test/opencode-coupling-audit.test.ts#L9-L13)). README가 선언한 멀티 하네스 개편([README.md#L11-L14](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/README.md#L11-L14))의 실행 형태다.

**두 에디션 = 같은 코어, 다른 어댑터.** Light 에디션(omo-codex)은 Codex CLI 플러그인 매니페스트로 skills/hooks/MCP 서버를 선언하는 별도 패키징이고([plugin.json#L21-L23](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-codex/plugin/.codex-plugin/plugin.json#L21-L23)), rules·comment-checker·git-bash·lsp·ultrawork·ulw-loop 등 이식 가능한 컴포넌트만 담는다([README.md#L117](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/README.md#L117)). 에이전트 오케스트레이션은 의도적으로 제외 — 베이스 하네스가 이미 제공하는 표면과 겹치는 부분은 들고 가지 않는다는 에디션 전략이다.

**확장은 전부 베이스의 훅 계약 안에서.** omo는 opencode를 포크하지 않고 기준선이 정의한 `(input, output)` 뮤테이션 훅([opencode.md 아키텍처](opencode.md#아키텍처))만으로 위 기능 전부를 구현한다. 내부적으로는 54종 훅을 이름 있는 단위로 쪼개 enum으로 고정하고([hooks.ts#L3-L60](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/config/schema/hooks.ts#L3-L60)), 사용자가 개별 훅을 켜고 끌 수 있게 한다 — 베이스의 훅 9개 정도의 확장점을 받아 54개의 기능 토글로 증폭시키는 구조다. 설정 주입도 같은 패턴이다: `config` 훅 안에서 에이전트 정의·MCP·권한을 선언적으로 재작성한다([plugin-interface.ts#L79](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/plugin-interface.ts#L79), [mcp-config-handler.ts#L50-L68](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/plugin-handlers/mcp-config-handler.ts#L50-L68)).

**개명의 잔재가 배포면에 남아 있다.** 레포는 oh-my-openagent로 개명됐지만 npm 패키지명·schema·바이너리 패키지(`oh-my-opencode-darwin-arm64` 등 플랫폼별 12종 내장 패키지 디렉토리)는 구명칭을 유지하고, 5개 bin 별칭이 전부 같은 진입점을 가리킨다([package.json#L24-L30](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/package.json#L24-L30)). 호환성을 패키지명 동결로 사는 선택이다.

## 모델별 튜닝

framework.md 정의: [모델별 튜닝](../framework.md#5-모델별-튜닝-per-model-tuning).

**메인 에이전트 프롬프트가 모델별 네이티브 변형으로 존재한다.** sisyphus 디렉토리에는 default(Claude·범용), claude-opus-4-7(literal-instruction 튜닝), claude-opus-4-8(silence-default + autonomy 튜닝), claude-fable-5, gemini(공격적 성향 교정 오버레이), gpt-5-4(블록 구조 가이드), gpt-5-5(Codex풍 섹션), kimi-k2-6 변형이 각각 별도 파일로 산다([sisyphus/index.ts#L1-L12](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/agents/sisyphus/index.ts#L1-L12)). 팩토리가 모델 ID 판별 함수(`isKimiK2Model`, `isGpt5_5Model`, `isClaudeOpus48Model` …)로 분기해 해당 변형을 고른다([sisyphus-agent-factory.ts#L45-L91](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/agents/sisyphus-agent-factory.ts#L45-L91)). 기준선의 모델별 프롬프트 라우팅([opencode.md 모델별 튜닝](opencode.md#모델별-튜닝))이 "프롬프트 파일 선택"이라면, omo는 에이전트 페르소나 수준에서 같은 일을 더 깊게 한다.

**카테고리 기반 모델 라우팅.** 작업 종류를 8개 빌트인 카테고리(visual-engineering, ultrabrain, deep, artistry, quick, unspecified-low/high, writing)로 나누고([categories.ts#L30-L39](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/config/schema/categories.ts#L30-L39)), 카테고리마다 모델·variant·thinking 예산·reasoningEffort·온도까지 통째로 설정한다([categories.ts#L4-L28](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/config/schema/categories.ts#L4-L28)). 빌트인 정의 자체가 Google/OpenAI/Anthropic/Kimi 모델사별 정의의 합성이다([builtin-categories.ts#L8-L13](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/delegate-task/builtin-categories.ts#L8-L13)). 모델 선택이 사용자 결정이 아니라 "작업 분류 → 모델"의 데이터 테이블이 된다.

**에이전트별 모델 폴백 체인.** 하네스 중립 패키지 model-core에 에이전트마다 (프로바이더 집합, 모델, variant) 우선순위 리스트가 박혀 있다: sisyphus는 claude-opus-4-7(max) → kimi-k2.6 → … → glm-5 → big-pickle 순으로 7단계, 탐색·사서 에이전트(explore/librarian)는 gpt-5.4-mini-fast나 minimax 같은 저비용 모델 체인을 쓴다([agent-model-requirements.ts#L3-L31](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/model-core/src/agent-model-requirements.ts#L3-L31), [#L62-L85](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/model-core/src/agent-model-requirements.ts#L62-L85)). "어떤 일에 어떤 두뇌를 쓰는가"가 역할(에이전트)과 작업 분류(카테고리) 양쪽에 이중으로 코드화되어 있다.

**에이전트-모델 결합의 강제.** 모델별 튜닝이 권고를 넘어 차단으로 간다: sisyphus를 비대응 GPT 모델로 돌리려 하면 "NEVER Use Sisyphus with GPT" 토스트를 띄우고 메시지의 에이전트를 hephaestus로 강제 교체한다([no-sisyphus-gpt/hook.ts#L11-L16](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/no-sisyphus-gpt/hook.ts#L11-L16), [#L72-L79](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/no-sisyphus-gpt/hook.ts#L72-L79)). 역방향 가드(no-hephaestus-non-gpt)도 훅 enum에 존재한다([hooks.ts#L40-L41](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/config/schema/hooks.ts#L40-L41)).

**지시문도 모델별로 변형된다.** ultrawork 주입 메시지는 플래너/GPT/Gemini/기본(Claude 최적화) 4계열로 라우팅된다([ultrawork/index.ts#L1-L9](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/keyword-detector/ultrawork/index.ts#L1-L9), [#L39-L56](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/keyword-detector/ultrawork/index.ts#L39-L56)). gemini/minimax 모델은 자동으로 unstable로 분류되어 루프 정책(강제 백그라운드 감시)까지 달라진다([categories.ts#L24-L25](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/config/schema/categories.ts#L24-L25)).
