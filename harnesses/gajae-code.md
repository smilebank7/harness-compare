- **분석 축:** 독립형 외부 coding-agent harness. Claude Code·Codex CLI·OpenCode·Claw Code 안에 숨어 들어가는 플러그인이 아니라, 선택한 repo/worktree 옆에서 `gjc` 프로세스, `.gjc` 상태, workflow skill, role agent, tmux worker, goal ledger를 직접 소유하는 구조에 집중한다.

## 개요

Gajae-Code는 README 첫 화면에서 “interviews, reviewed plans, tmux-native execution, and durable verification”을 수행하는 coding-agent runner라고 설명하며([README.md#L5-L10](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/README.md#L5-L10)), 동시에 beta-stage라서 출력 검증을 요구한다고 못박는다([README.md#L22-L22](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/README.md#L22-L22)). 핵심 자기정의는 “external coding-agent harness”이며, 공개 workflow 표면을 `deep-interview -> ralplan -> ultragoal`과 선택적 `team` 실행으로 그린다([README.md#L24-L33](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/README.md#L24-L33)). 설치·실행 표면은 `bun install -g gajae-code`, scoped package `@gajae-code/coding-agent`, 그리고 bin `gjc`로 정리된다([README.md#L35-L41](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/README.md#L35-L41), [packages/coding-agent/package.json#L28-L32](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/package.json#L28-L32)).

README는 `gjc`, `gjc --tmux`, `gjc --tmux --worktree`를 빠른 시작 명령으로 두고, 세션 안 workflow는 `/skill:deep-interview`, `/skill:ralplan`, `gjc ultragoal create-goals`, `gjc ultragoal complete-goals`로 예시한다([README.md#L43-L65](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/README.md#L43-L65)). 공개 workflow skill은 deep-interview, ralplan, ultragoal, team 네 개이고([README.md#L75-L84](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/README.md#L75-L84)), role agent는 executor, architect, planner, critic 네 개로 제한된다([README.md#L86-L95](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/README.md#L86-L95)). AGENTS.md도 같은 네 skill과 네 role agent를 source path까지 표로 고정하고, repo-visible `.gjc` default copy가 아니라 source-bundled definition을 SSOT로 둔다([AGENTS.md#L5-L30](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/AGENTS.md#L5-L30)).

플러그인형 하네스와의 차이는 README의 boundary 표에 드러난다. GJC는 Codex CLI, Claude Code, OpenCode, Claw Code 옆에서 실행되며 각각의 extension으로 들어가지 않는다고 설명한다([README.md#L97-L106](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/README.md#L97-L106)). AGENTS.md의 routing도 낮은 위험 직접 구현, deep-interview, ralplan, ultragoal, team 순으로 workflow를 고르고, deep-interview/ralplan에서 사용자 승인 없이 구현으로 넘어가지 말라고 규정한다([AGENTS.md#L32-L44](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/AGENTS.md#L32-L44)). 따라서 GJC의 기본 단위는 “다른 에이전트 런타임에 주입되는 skill 묶음”이 아니라, **외부 runner가 repo state와 workflow state를 들고 계획·검토·실행·검증 권위를 단계적으로 이동시키는 독립형 하네스**다.

## 하네스 엔지니어링

framework.md 정의: [하네스 엔지니어링](../framework.md#1-하네스-엔지니어링-harness-engineering).

**도구 표면은 GJC 프로세스가 직접 구성한다.** `tools/index.ts`는 Ask, AST Grep/Edit, Bash, Browser, Debug, Eval, Find, GitHub, IRC, Job, Monitor, Read, Resolve, Search, Skill, Subagent, Todo, Write와 LSP, Goal, Task, WebSearch 계열을 한 파일에서 모아 세션 도구 집합으로 만든다([packages/coding-agent/src/tools/index.ts#L32-L61](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/index.ts#L32-L61)). 공개 callable map도 `read`, `bash`, `edit`, `ast_grep`, `ast_edit`, `ask`, `debug`, `eval`, `find`, `search`, `lsp`, `browser`, `task`, `subagent`, `job`, `monitor`, `irc`, `todo_write`, `web_search`, `write`, `skill`, `goal` 등을 직접 생성한다([packages/coding-agent/src/tools/index.ts#L313-L348](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/index.ts#L313-L348)). 기본 essential 도구는 `read`, `bash`, `edit`이며 설정 override가 없을 때 이 세 개가 기본값이다([packages/coding-agent/src/tools/index.ts#L289-L304](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/index.ts#L289-L304)). 이는 Claude Code plugin manifest나 OpenCode plugin hook이 아니라 GJC 내부 ToolSession의 public surface다.

**ToolSession은 도구 호출에 필요한 repo·skill·goal·model·subagent 문맥을 한 구조체로 묶는다.** `ToolSession`에는 cwd, UI 여부, context file, workspace tree, skills, active skill state/phase, LSP 활성화, event bus, output schema, task depth/current agent, session file, artifact manager, model/auth/settings, plan mode, goal mode/runtime, client bridge, compact context, todo phases, MCP/tool discovery, tool-choice queue가 들어간다([packages/coding-agent/src/tools/index.ts#L119-L240](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/index.ts#L119-L240)). 같은 구조체는 custom message dispatch와 fork-context seed builder도 담아 skill handoff와 subagent 실행이 같은 세션 권한 모델을 공유하게 한다([packages/coding-agent/src/tools/index.ts#L264-L283](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/index.ts#L264-L283)).

**Read 도구는 단순 파일 열기가 아니라 context 공급 장치다.** `read.ts`는 sqlite, native glob/code summary, image metadata, notebook conversion, archive reader, URL fetch, conflict detection, sqlite reader, workspace tree, document conversion을 함께 가져온다([packages/coding-agent/src/tools/read.ts#L1-L90](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/read.ts#L1-L90)). PDF/DOC/PPT/XLS/RTF/EPUB는 markit 변환 대상으로 분리되고, 코드 summary에는 byte·line 상한이 있다([packages/coding-agent/src/tools/read.ts#L92-L105](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/read.ts#L92-L105)). 명시 범위 읽기는 line number 또는 hashline을 붙일 수 있고([packages/coding-agent/src/tools/read.ts#L111-L124](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/read.ts#L111-L124)), summary elision footer는 raw/range 재읽기 방법을 모델에게 알려 준다([packages/coding-agent/src/tools/read.ts#L183-L194](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/read.ts#L183-L194)). 따라서 GJC의 read는 “파일 내용”을 넘기는 API가 아니라, 모델이 편집·검토 가능한 형태로 세계를 재구성하는 하네스 장치다.

**Edit 표면은 여러 문법과 LSP writethrough를 같은 도구 안에 묶는다.** `edit/index.ts`는 replace, patch, hashline, vim, apply_patch schema와 grammar를 함께 등록하고([packages/coding-agent/src/edit/index.ts#L1-L57](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/edit/index.ts#L1-L57)), fuzzy match 설정과 LSP format/diagnostics writethrough를 세션 설정으로 결정한다([packages/coding-agent/src/edit/index.ts#L72-L118](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/edit/index.ts#L72-L118)). apply_patch가 여러 파일을 건드릴 때는 파일별 결과를 모아 diff, diagnostics, firstChangedLine, old/new text를 집계하고([packages/coding-agent/src/edit/index.ts#L120-L194](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/edit/index.ts#L120-L194)), 다중 edit 중 일부가 실패하면 aggregate result를 error branch로 넘겨 proposed diff가 성공처럼 보이는 문제를 막는다([packages/coding-agent/src/edit/index.ts#L258-L272](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/edit/index.ts#L258-L272)).

**Hashline 편집은 anchor 무결성을 하네스 레벨에서 강제한다.** `hashline/apply.ts`는 모든 anchor hash를 검증하며 mismatch가 있으면 auto-rebase하지 않고 모델이 다시 읽고 re-anchor하도록 `HashlineMismatchError`로 표면화한다고 주석과 함수가 명시한다([packages/coding-agent/src/hashline/apply.ts#L45-L66](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/hashline/apply.ts#L45-L66)). 같은 파일은 edit를 target line별로 bucket하고 replacement group을 찾는 로직을 둔다([packages/coding-agent/src/hashline/apply.ts#L95-L143](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/hashline/apply.ts#L95-L143)). 단일 라인 중복 흡수는 구조적 닫힘 delimiter에만 허용되며, bracket counter가 안전한 방향으로 실패하도록 제한된다([packages/coding-agent/src/hashline/apply.ts#L175-L200](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/hashline/apply.ts#L175-L200)). 이것은 모델 프롬프트가 아니라 edit tool contract 자체가 stale-write 위험을 줄이는 설계다.

**Bash 도구는 터미널 실행 표면과 안전 장치를 같이 제공한다.** `bash.ts`는 AsyncJobManager, bash executor, GJC runtime env, restricted role-agent bash, internal URL expansion, streaming tail output, command fixups, interceptor, pty selection, timeout clamp를 함께 가져온다([packages/coding-agent/src/tools/bash.ts#L1-L34](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/bash.ts#L1-L34)). 스키마는 command/env/timeout/cwd/pty와 async 확장을 노출하고([packages/coding-agent/src/tools/bash.ts#L51-L61](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/bash.ts#L51-L61)), env 이름은 정규식으로 검증되어 display escape를 거친다([packages/coding-agent/src/tools/bash.ts#L115-L144](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/bash.ts#L115-L144)). timeout clamp notice도 별도 함수로 표면화된다([packages/coding-agent/src/tools/bash.ts#L217-L220](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/bash.ts#L217-L220)).

## 루프 엔지니어링

framework.md 정의: [루프 엔지니어링](../framework.md#2-루프-엔지니어링-loop-engineering).

**GJC의 요구사항 루프는 deep-interview에서 ambiguity gate로 시작한다.** Deep Interview skill은 질문으로 hidden assumption을 드러내고 weighted dimension ambiguity가 run threshold 아래로 내려갈 때까지 진행을 막는다고 목적을 설명한다([packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L13-L15](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L13-L15)). 실행 정책은 한 번에 하나의 질문, weakest dimension targeting, Round 0 topology gate, answer마다 ambiguity scoring, multi-component per-component scoring, explicit approval 전 실행 금지를 요구한다([packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L40-L55](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L40-L55)). threshold resolution은 user/project settings를 읽고 source를 상태와 최종 spec metadata에 보존해야 하며, threshold marker를 첫 user-visible line으로 요구한다([packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L75-L96](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L75-L96)).

**Round 0 topology gate는 depth-first 질문이 한 컴포넌트에 과적합되는 것을 막는 루프 장치다.** Deep Interview skill은 Phase 1 뒤 Phase 2 scoring 전에 scope shape를 lock하라고 설명하고, top-level verbs/nouns/workstreams/surfaces/integrations/deliverables를 1-6개 component로 추출하게 한다([packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L168-L176](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L168-L176)). 확인 뒤에는 component id/name/description/status/evidence/clarity scores/weakest dimension과 deferral을 state에 저장하는 JSON shape를 둔다([packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L190-L223](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L190-L223)). 질문 루프는 active component + dimension pair 중 가장 낮은 clarity score를 고르고, 동률일 때 component rotation을 하며, 질문 앞에서 왜 해당 pair가 bottleneck인지 밝히도록 한다([packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L237-L255](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L237-L255)).

**Ralplan 루프는 Planner → Architect → Critic consensus를 pending approval로 멈춘다.** Ralplan skill은 interactive가 없으면 Planner/Architect/Critic loop를 자동으로 돌리고 final plan을 pending approval로 표시한 뒤 실행 없이 멈춘다고 flag 설명에 둔다([packages/coding-agent/src/defaults/gjc/skills/ralplan/SKILL.md#L20-L27](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/ralplan/SKILL.md#L20-L27)). Planning/Execution Boundary는 명시 승인 전 mutation, source edit, commit, push, execution skill invocation, implementation delegation을 금지한다([packages/coding-agent/src/defaults/gjc/skills/ralplan/SKILL.md#L36-L46](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/ralplan/SKILL.md#L36-L46)). Consensus workflow는 Planner를 한 번 띄워 plan과 RALPLAN-DR summary를 만들고, Architect를 기다린 뒤 Critic을 실행하며, Critic이 approve하지 않으면 같은 Planner subagent를 resume하는 re-review loop를 최대 5회 반복한다([packages/coding-agent/src/defaults/gjc/skills/ralplan/SKILL.md#L56-L89](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/ralplan/SKILL.md#L56-L89)).

**Native ralplan runtime은 loop 실행자가 아니라 state/artifact persistence 경계다.** `ralplan-runtime.ts`는 native `gjc ralplan`이 documented flag surface를 검증하고 `.gjc/state/ralplan-state.json`을 seed하지만 Planner/Architect/Critic loop 자체는 bundled `/skill:ralplan`에 있다고 적는다([packages/coding-agent/src/gjc-runtime/ralplan-runtime.ts#L12-L29](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/gjc-runtime/ralplan-runtime.ts#L12-L29)).
