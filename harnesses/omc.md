## 개요

oh-my-claudecode는 자신을 "Multi-agent orchestration for Claude Code"라고 정의하고, Codex 사용자는 별도 oh-my-codex를 보라고 안내한다([README.md#L12-L16](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L12-L16)). 배포 표면은 둘로 갈린다. Claude Code 플러그인 경로는 `/plugin marketplace add`와 `/plugin install`을 전제로 하고([README.md#L52-L65](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L52-L65)), npm 경로는 `oh-my-claude-sisyphus` 패키지가 `oh-my-claudecode`, `omc`, `omc-cli` 세 bin을 노출한다([README.md#L67-L71](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L67-L71), [package.json#L14-L18](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/package.json#L14-L18)). 패키지 출하물에는 `agents`, `bridge`, `commands`, `hooks`, `scripts`, `skills`, `.claude-plugin`, `.mcp.json`이 함께 들어가므로, 이 레포는 단순 스킬 묶음이 아니라 Claude Code 플러그인·CLI 브리지·훅 런타임을 같이 배포하는 하네스다([package.json#L19-L40](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/package.json#L19-L40)).

레포가 표방하는 중심 표면은 Team이다. README는 `/team`을 Claude Code 세션 안 native team workflow로, `omc team`을 터미널에서 tmux CLI 워커를 띄우는 별도 런타임으로 구분하고([README.md#L105-L118](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L105-L118)), Team 파이프라인을 `team-plan → team-prd → team-exec → team-verify → team-fix`로 고정한다([README.md#L130-L142](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L130-L142)). 동시에 README는 Autopilot, Ralph, Ultrawork, Deep Interview는 터미널 CLI 하위 명령이 아니라 세션 안 스킬이라고 못박는다([README.md#L112-L118](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L112-L118)). 따라서 OMC의 설계 단위는 "Claude Code에 기능 하나를 더한다"가 아니라, **플러그인 훅·스킬·역할 에이전트·tmux 워커·상태 파일을 묶어 여러 실행 권위를 선택하게 하는 오케스트레이션 레이어**다.

## 하네스 엔지니어링

framework.md 정의: [하네스 엔지니어링](../framework.md#1-하네스-엔지니어링-harness-engineering).

**플러그인 매니페스트가 스킬·MCP·커맨드 표면의 루트다.** `.claude-plugin/plugin.json`은 플러그인 이름과 설명을 선언한 뒤 40개 안팎의 `skills/*` 경로, `.mcp.json`, `commands/`를 한 번에 노출한다([.claude-plugin/plugin.json#L1-L62](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/.claude-plugin/plugin.json#L1-L62)). 이 목록에는 `deep-interview`, `ralplan`, `team`, `ultragoal`, `ultrawork`, `ask`, `debug`, `wiki`가 같이 들어간다([.claude-plugin/plugin.json#L18-L58](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/.claude-plugin/plugin.json#L18-L58)). 모델이 호출할 수 있는 세션 스킬과 커맨드의 상한선을 매니페스트가 정하므로, 이는 프롬프트 내용 이전의 도구·컨텍스트 표면이다.

**훅 표면은 Claude Code 이벤트 전반을 덮는다.** `hooks/hooks.json`은 UserPromptSubmit에 `keyword-detector.mjs`와 `skill-injector.mjs`, SessionStart에 세션 시작·프로젝트 메모리·위키 시작 훅, PreToolUse에 `pre-tool-enforcer.mjs`, PermissionRequest(Bash)에 `permission-handler.mjs`, PostToolUse에 검증·메모리·규칙 주입 훅을 연결한다([hooks/hooks.json#L4-L108](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/hooks/hooks.json#L4-L108)). 같은 매니페스트는 SubagentStart/Stop 추적, PreCompact 전처리, Stop 훅의 context guard·persistent mode·code simplifier, SessionEnd 위키 종료까지 선언한다([hooks/hooks.json#L121-L210](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/hooks/hooks.json#L121-L210)). 단일 턴 전·후와 세션 경계 모두에 커맨드 훅을 심는 이 구조가 OMC의 하네스 기본면이다.

**도구 사용 전 게이트는 모델·환경·안티 슬롭 규칙까지 본다.** `pre-tool-enforcer.mjs`는 provider-specific model id, Bedrock/Vertex 환경, tier alias, subagent-safe model 검증을 자체 함수로 갖고([scripts/pre-tool-enforcer.mjs#L21-L123](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/pre-tool-enforcer.mjs#L21-L123)), 에이전트 정의의 YAML frontmatter에서 `model:` 필드만 추출하도록 별도 파서를 둔다([scripts/pre-tool-enforcer.mjs#L124-L160](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/pre-tool-enforcer.mjs#L124-L160)). 같은 파일은 Task/Bash/Edit/Write 등 여러 도구를 "slop risk" 표면으로 분류하고, fallback/workaround 문구가 실제 회피 구현 의도인지 판단하는 패턴과 문서·자기참조 예외를 둔다([scripts/pre-tool-enforcer.mjs#L164-L196](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/pre-tool-enforcer.mjs#L164-L196)). 즉 권한 게이트가 단순 허용/거부가 아니라 모델 라우팅 가능성, 역할 파일, 작업 의도를 함께 읽는다.

**서브에이전트 표면은 역할 파일 + 훅 검증으로 구성된다.** `agents/explore.md`는 haiku·read-only 탐색 역할을 선언하고([agents/explore.md#L1-L14](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/explore.md#L1-L14)), `agents/executor.md`는 sonnet 구현 역할과 검증 책임을 선언한다([agents/executor.md#L1-L15](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/executor.md#L1-L15)). `agents/architect.md`와 `agents/critic.md`는 opus·read-only 검토/비평 역할로 묶인다([agents/architect.md#L1-L7](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/architect.md#L1-L7), [agents/critic.md#L1-L7](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/critic.md#L1-L7)). SubagentStop에는 `verify-deliverables.mjs`가 붙어 완료 에이전트가 산출물 파일을 실제로 냈는지 프로젝트·기본 deliverables 설정으로 확인하되, 경고만 추가하는 non-blocking 훅으로 동작한다([scripts/verify-deliverables.mjs#L3-L21](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/verify-deliverables.mjs#L3-L21), [#L146-L236](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/verify-deliverables.mjs#L146-L236)).

**tmux CLI 워커도 별도 하네스 표면이다.** Team 런타임은 `CliAgentType`을 `claude | codex | gemini | cursor | grok`으로 제한하고([src/team/model-contract.ts#L8-L20](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/model-contract.ts#L8-L20)), CLI binary 경로가 `/tmp`·`/var/tmp`·`/dev/shm` 같은 위치에서 해석되면 거부하며 신뢰 prefix를 검증한다([src/team/model-contract.ts#L51-L145](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/model-contract.ts#L51-L145)). Claude 워커는 `--dangerously-skip-permissions`와 API-key 환경에서 `--bare`를 붙이고, Codex 워커는 `--dangerously-bypass-approvals-and-sandbox`로 띄우며 prompt mode를 쓰지 않는다([src/team/model-contract.ts#L172-L219](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/model-contract.ts#L172-L219)). 이는 Claude Code 세션 내부 Task가 아니라 외부 CLI 프로세스의 권한·실행 인자를 OMC가 직접 조립하는 하네스다.

## 루프 엔지니어링

framework.md 정의: [루프 엔지니어링](../framework.md#2-루프-엔지니어링-loop-engineering).

**Team은 단계·검증·수정 반복을 공개 계약으로 둔다.** README는 Team 실행을 `team-plan → team-prd → team-exec → team-verify → team-fix (loop)`로 설명하고([README.md#L140-L142](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L140-L142)), `skills/team/SKILL.md`도 같은 파이프라인과 stage별 agent routing을 명시한다([skills/team/SKILL.md#L98-L121](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/skills/team/SKILL.md#L98-L121)). Team 스킬의 종료 규칙은 `team-exec -> team-verify -> team-fix`를 검증 통과 또는 terminal blocked/failed까지 반복하고, fix 시도는 설정 한도에 묶는다는 것이다([skills/team/SKILL.md#L147-L155](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/skills/team/SKILL.md#L147-L155)). 여기서 핵심은 어떤 도구가 있느냐가 아니라, 실행 흐름이 시간 축에서 어떤 조건으로 다음 단계에 들어가고 되돌아오는가다.

**팀 런타임은 작업 파일 상태와 worker 생사를 루프 상태로 쓴다.** `src/team/runtime.ts`의 task record는 `pending | in_progress | completed | failed`, owner, assigned/completed/failed 시각을 가진다([src/team/runtime.ts#L92-L104](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/runtime.ts#L92-L104)). 상태 루트는 작업 디렉터리 아래 `.omc/state/team/<teamName>`이고([src/team/runtime.ts#L111-L119](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/runtime.ts#L111-L119)), task claim은 lock 안에서 pending만 in_progress로 바꾼다([src/team/runtime.ts#L202-L214](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/runtime.ts#L202-L214)). worker가 죽었을 때의 transition도 `requeued | failed | skipped` 액션으로 모델링된다([src/team/runtime.ts#L251-L263](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/runtime.ts#L251-L263)). 루프 상태가 대화 기억이 아니라 파일·lock·pane 생사로 내려온다.

**Stop 훅의 persistent-mode는 여러 실행 모드의 계속/정지 판단을 중앙화한다.** `persistent-mode.mjs`는 지원 모드를 ralph, ultragoal, autopilot, ultrapilot, swarm, ultrawork, ultraqa, pipeline, team으로 열거한다([scripts/persistent-mode.mjs#L3-L9](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/persistent-mode.mjs#L3-L9)). 메인 루프는 context-limit stop, critical context, user abort, auth failure, scheduled wakeup, session cancel, pending async work를 먼저 통과시켜 재귀·컴팩션 교착을 피한다([scripts/persistent-mode.mjs#L949-L999](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/persistent-mode.mjs#L949-L999), [#L1057-L1060](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/persistent-mode.mjs#L1057-L1060)). 이후 세션·프로젝트에 맞는 mode state들을 읽고([scripts/persistent-mode.mjs#L1001-L1055](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/persistent-mode.mjs#L1001-L1055)), 예컨대 Ralph는 iteration이 max 아래면 state를 갱신하고 `decision: "block"`으로 "Work is NOT done" reason을 반환한다([scripts/persistent-mode.mjs#L1071-L1108](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/persistent-mode.mjs#L1071-L1108)).

**PostToolUse는 검증 리마인더와 컨텍스트 압박 감지를 루프 보조 신호로 만든다.** `post-tool-verifier.mjs`는 도구 호출 통계, compaction warning threshold, 대용량 출력 도구 집합, 세션별 상태 파일을 관리한다([scripts/post-tool-verifier.mjs#L20-L34](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/post-tool-verifier.mjs#L20-L34), [#L130-L187](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/post-tool-verifier.mjs#L130-L187)). `.omc` 루트 해석은 workspace marker, git root, fallback 순서로 수행된다([scripts/post-tool-verifier.mjs#L36-L94](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/post-tool-verifier.mjs#L36-L94)). 이 훅이 직접 작업을 반복시키는 것은 아니지만, 도구 사용 후 검증·컴팩션·세션 상태를 다음 턴에 알려 루프의 guard rail로 쓰인다.

**완료 주장의 검증은 코드 훅과 스킬 계약이 나눠 가진다.** Deliverable hook은 산출물 파일 존재·최소 크기·필수 섹션·패턴을 확인하고, 이슈가 있으면 SubagentStop additionalContext로 경고한다([scripts/verify-deliverables.mjs#L95-L144](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/verify-deliverables.mjs#L95-L144), [#L179-L231](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/verify-deliverables.mjs#L179-L231)). Team 스킬은 각 단계가 handoff 문서를 남기고 다음 단계가 그것을 읽도록 요구한다([skills/team/SKILL.md#L156-L181](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/skills/team/SKILL.md#L156-L181)). 코드 훅은 최소 산출물 유무를 보조하고, 스킬 프롬프트는 검증·수정 반복과 handoff 의미를 규정한다.

## 프롬프트 엔지니어링

framework.md 정의: [프롬프트 엔지니어링](../framework.md#3-프롬프트-엔지니어링-prompt-engineering).

**스킬 본문이 세션 행동의 주된 지시문이다.** 패키지는 `skills/` 디렉터리를 번들 SSOT로 읽고, frontmatter·alias·pipeline metadata·runtime guidance를 합성해 skill entry를 만든다([src/features/builtin-skills/skills.ts#L1-L20](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/builtin-skills/skills.ts#L1-L20), [#L231-L263](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/builtin-skills/skills.ts#L231-L263)). `plan`, `review`, `security-review` 같은 Claude Code native command 이름은 OMC skill short name이 shadow하지 않도록 `omc-` prefix로 안전화한다([src/features/builtin-skills/skills.ts#L54-L85](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/builtin-skills/skills.ts#L54-L85)). 같은 로더는 deep-interview 템플릿에 threshold placeholder를 런타임 설정으로 치환한다([src/features/builtin-skills/skills.ts#L184-L228](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/builtin-skills/skills.ts#L184-L228)). 즉 프롬프트는 정적 Markdown이지만, 렌더링 시점에 설정과 이름 충돌 규칙이 섞인다.

**키워드 감지는 자연어 프롬프트를 모드 지시문으로 바꾼다.** `keyword-detector.mjs`는 cancelomc/stopomc, ralph, autopilot, ultrawork/ulw, ccg, ralplan, deep interview, ai-slop-cleaner, tdd, code review, security review, ultrathink, deepsearch, analyze를 우선순위로 나열한다([scripts/keyword-detector.mjs#L3-L24](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/keyword-detector.mjs#L3-L24)). 같은 파일은 `<think-mode>`, `<search-mode>`, `<analyze-mode>`, `<tdd-mode>`, `<code-review-mode>`, `<security-review-mode>` 문자열을 코드 안에 들고 있다([scripts/keyword-detector.mjs#L114-L183](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/keyword-detector.mjs#L114-L183)). 오탐 방지를 위해 code block, XML tag, URL, file path, review seed table을 걷어내는 sanitizer도 들어간다([scripts/keyword-detector.mjs#L212-L263](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/keyword-detector.mjs#L212-L263)).

**Skill injector는 사용자·프로젝트 skill을 자동 컨텍스트로 끌어온다.** `skill-injector.mjs`는 UserPromptSubmit 훅에서 prompt trigger에 맞는 learned skill을 주입한다고 설명하고([scripts/skill-injector.mjs#L3-L11](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/skill-injector.mjs#L3-L11)), hook payload의 session id를 우선시하며 `.omc-workspace`나 `.git`을 따라 `.omc` 루트를 찾는다([scripts/skill-injector.mjs#L30-L114](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/skill-injector.mjs#L30-L114)). README도 `.omc/skills/`는 프로젝트 scope, `~/.omc/skills/`는 user scope이며 trigger가 맞으면 자동 주입된다고 설명한다([README.md#L286-L311](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L286-L311)). 코드가 파일을 찾는 장치는 하네스지만, 주입되는 skill Markdown과 trigger 설명은 프롬프트 차원의 설계다.

**역할 에이전트 프롬프트는 페르소나·금지·성공 기준을 frontmatter와 본문으로 고정한다.** Architect는 "Strategic Architecture & Debugging Advisor", Opus, read-only로 정의되고 근거 없는 조언 금지·file:line 근거를 요구한다([agents/architect.md#L1-L18](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/architect.md#L1-L18), [#L20-L36](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/architect.md#L20-L36)). Critic은 false approval 비용과 gap analysis를 전면에 두고, CRITICAL/MAJOR 근거와 realist check를 요구한다([agents/critic.md#L9-L43](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/critic.md#L9-L43)). Executor는 작은 diff, LSP diagnostics, build/test 검증, 임시 코드 금지를 성공 기준으로 둔다([agents/executor.md#L20-L40](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/executor.md#L20-L40)). 역할 파일 자체가 모델에게 주입되는 자연어 계약이다.

**Team 스킬은 오케스트레이션 자체를 프롬프트 시퀀스로 기술한다.** `skills/team/SKILL.md`는 사용법, agent count, stage routing, entry/exit criteria, verify/fix loop, handoff convention까지 문서로 선언한다([skills/team/SKILL.md#L15-L29](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/skills/team/SKILL.md#L15-L29), [#L123-L181](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/skills/team/SKILL.md#L123-L181)). 같은 파일은 native team과 tmux CLI worker를 모두 설명하지만, 사용자 `N:agent-type`이 team-exec worker만 덮고 나머지는 stage별 specialist가 고른다는 규칙을 텍스트로 못박는다([skills/team/SKILL.md#L104-L121](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/skills/team/SKILL.md#L104-L121)). 이는 코드 없는 정책이 아니라, 모델이 그 텍스트를 읽고 실행 단계를 고르는 프롬프트 프로그램이다.

## 아키텍처

framework.md 정의: [아키텍처](../framework.md#4-아키텍처-architecture).

**한 레포가 플러그인, npm CLI, 브리지, 훅, 스킬을 함께 배포한다.** `package.json`의 build 스크립트는 TypeScript 빌드 뒤 skill bridge, MCP server, bridge entry, runtime CLI, team server, CLI를 차례로 생성한다([package.json#L41-L50](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/package.json#L41-L50)). 출하 파일 목록은 `dist`, `agents`, `bridge`, `commands`, `hooks`, `scripts`, `skills`, `templates`, `docs`, `.claude-plugin`, `.mcp.json`까지 포함한다([package.json#L19-L40](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/package.json#L19-L40)). lazycodex처럼 upstream 한 줄 설치를 감싼 배포판이 아니라, Claude Code 플러그인 런타임의 여러 층을 한 패키지 안에서 빌드·배포하는 구조다.

**세션 내 workflow와 터미널 runtime이 명시적으로 분리된다.** README의 CLI vs in-session 표는 `omc setup`과 `/setup`, `omc ask`와 `/ask`, `omc team`과 `/team`을 각각 대응시키면서 Team의 두 런타임이 다르다고 적는다([README.md#L105-L118](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L105-L118)). `omc team`은 tmux CLI workers이고, `/team`은 Claude Code in-session native team workflow다([README.md#L138-L142](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L138-L142)). 이 분리는 기능명의 중복이 아니라 상태·통신·프로세스 모델이 갈라지는 아키텍처 경계다.

**상태 루트는 `.omc`와 workspace marker 규칙으로 묶인다.** README는 여러 repo가 부모 디렉터리를 공유할 때 `.omc-workspace` marker를 두면 하위 repo들이 같은 `.omc/` state root를 쓴다고 설명한다([README.md#L315-L328](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L315-L328)). hook 스크립트들도 workspace marker와 git root를 따라 `.omc` 루트를 찾는 로직을 중복 보유한다([scripts/post-tool-verifier.mjs#L36-L94](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/post-tool-verifier.mjs#L36-L94), [scripts/skill-injector.mjs#L80-L114](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/skill-injector.mjs#L80-L114)). Team 런타임의 상태도 cwd 아래 `.omc/state/team/<teamName>`에 놓인다([src/team/runtime.ts#L115-L119](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/runtime.ts#L115-L119)). 상태 위치는 단일 기능 파일이 아니라 전체 하네스의 작업공간 경계를 정하는 설계다.

**Team 아키텍처는 native team과 tmux worker를 병렬 표면으로 둔다.** Team 스킬 앞부분은 Claude Code native team tools, inter-agent messaging, task dependencies를 설명하고([skills/team/SKILL.md#L9-L13](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/skills/team/SKILL.md#L9-L13)), 아키텍처 예시는 TeamCreate, TaskCreate, TaskUpdate, Task spawns, monitor loop, TeamDelete 흐름을 ASCII로 제시한다([skills/team/SKILL.md#L45-L78](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/skills/team/SKILL.md#L45-L78)). 반면 README의 tmux CLI workers 절은 `omc team N:codex/gemini/grok/claude`가 split-pane 실제 CLI 프로세스를 띄우고, 작업이 끝나면 워커가 죽는다고 설명한다([README.md#L156-L184](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L156-L184)). 한 레포 안에 두 병렬 실행 골격이 있고, 공통 이름 Team이 그 위를 덮는다.

**역할 라우팅은 stage 정책과 regex 라우터가 같이 담당한다.** Team 스킬은 stage별 필수·선택 agent를 표로 선언하고, risk나 file count에 따른 verify escalation도 텍스트로 둔다([skills/team/SKILL.md#L104-L121](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/skills/team/SKILL.md#L104-L121)). 코드 쪽 `role-router.ts`는 build-fix, debug, docs, design, cleanup, review, verification, implementation intent를 regex로 판정하고, security domain 여부에 따라 security-reviewer 또는 fallback role을 고른다([src/team/role-router.ts#L31-L127](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/role-router.ts#L31-L127), [#L150-L243](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/role-router.ts#L150-L243)). 큰 골격은 스킬 문서, 세부 task-to-role 자동화는 TypeScript 라우터가 맡는 분업이다.

**브리지 산출물이 아키텍처 결합점을 노출한다.** npm 패키지는 `bridge/mcp-server.cjs`, `bridge/team-bridge.cjs`, `bridge/team-mcp.cjs`, `bridge/runtime-cli.cjs`, `bridge/cli.cjs`를 명시적으로 출하한다([package.json#L23-L29](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/package.json#L23-L29)). build 스크립트도 bridge entry와 team server, runtime CLI 생성을 별도 단계로 둔다([package.json#L41-L47](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/package.json#L41-L47)). 이 문서는 생성된 CJS 내부를 세부 분석하지 않았지만, `bridge/`가 Claude Code plugin runtime과 터미널 `omc` CLI 사이의 결합층이라는 점은 패키지 경계만으로도 확인된다.

## 모델별 튜닝

framework.md 정의: [모델별 튜닝](../framework.md#5-모델별-튜닝-per-model-tuning).

**작업 복잡도 라우터는 LOW/MEDIUM/HIGH tier를 모델 타입에 매핑한다.** `routeTask`는 forceInherit, disabled config, explicit model, agent override를 먼저 처리하고, 그 다음 task prompt에서 신호를 추출해 rules와 score를 결합한다([src/features/model-routing/router.ts#L22-L64](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/model-routing/router.ts#L22-L64)). rules와 scorer가 2단계 이상 벌어지면 confidence를 낮추고 높은 tier를 택해 under-provisioning을 피한다([src/features/model-routing/router.ts#L71-L104](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/model-routing/router.ts#L71-L104)). 최종 decision은 tier model과 modelType을 반환하며, modelType→tier 변환은 opus=HIGH, haiku=LOW, sonnet=MEDIUM으로 고정된다([src/features/model-routing/router.ts#L106-L149](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/model-routing/router.ts#L106-L149)).

**신호 추출은 prompt의 표면 특징과 작업 구조를 분리한다.** `signals.ts`는 lexical 신호로 word count, file path count, code block count, architecture/debug/simple/risk keyword, question depth, implicit requirement를 뽑고([src/features/model-routing/signals.ts#L17-L36](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/model-routing/signals.ts#L17-L36)), structural 신호로 estimated subtasks, cross-file dependency, test requirement, domain, external knowledge, reversibility, impact scope를 뽑는다([src/features/model-routing/signals.ts#L38-L53](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/model-routing/signals.ts#L38-L53)). vague pattern, file path, list item, cross-file indicator, test indicator까지 모두 regex로 구현되어 있다([src/features/model-routing/signals.ts#L85-L213](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/model-routing/signals.ts#L85-L213)). 모델 선택은 "사용자가 어떤 모델명을 썼나"만이 아니라, 요청의 형태가 어떤 비용·추론 tier를 요구하는지에 대한 정적 신호 계산이다.

**카테고리별 위임은 tier뿐 아니라 temperature와 thinking budget도 조정한다.** `delegation-categories/index.ts`는 visual-engineering, ultrabrain, artistry, quick, writing, unspecified-low/high 카테고리에 tier·temperature·thinkingBudget·promptAppend를 배정한다([src/features/delegation-categories/index.ts#L31-L82](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/delegation-categories/index.ts#L31-L82)). 예컨대 ultrabrain은 HIGH, temperature 0.3, max thinking budget, quick은 LOW, temperature 0.1, low budget이다([src/features/delegation-categories/index.ts#L42-L62](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/delegation-categories/index.ts#L42-L62)). 카테고리 감지는 키워드 매치가 2개 이상일 때만 인정한다([src/features/delegation-categories/index.ts#L94-L130](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/delegation-categories/index.ts#L94-L130), [#L179-L223](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/delegation-categories/index.ts#L179-L223)).

**역할 파일 자체에도 모델 tier가 박혀 있다.** explore는 haiku, executor는 sonnet, architect와 critic은 opus로 선언된다([agents/explore.md#L1-L7](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/explore.md#L1-L7), [agents/executor.md#L1-L6](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/executor.md#L1-L6), [agents/architect.md#L1-L7](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/architect.md#L1-L7), [agents/critic.md#L1-L7](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/critic.md#L1-L7)). Team 스킬도 stage routing 표에서 explore(haiku), planner/analyst/architect/critic(opus), executor/debugger/designer/test-engineer(sonnet), code-reviewer(opus) 같은 조합을 노출한다([skills/team/SKILL.md#L104-L121](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/skills/team/SKILL.md#L104-L121)). 모델 선택이 실행 시점 heuristic만이 아니라 역할 정의와 스킬 단계표에도 중복 기록된다.

**멀티 CLI 워커는 모델별 실행 인자를 별도로 둔다.** Claude worker는 Claude Code alias normalization과 provider-specific model id 예외를 갖고([src/team/model-contract.ts#L188-L201](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/model-contract.ts#L188-L201)), Codex worker는 JSONL assistant/result 메시지 파서를 둔다([src/team/model-contract.ts#L207-L237](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/model-contract.ts#L207-L237)). Gemini는 `--approval-mode yolo`와 `-p` prompt mode를 쓰고, Grok은 `--always-approve`와 prompt mode를 쓴다([src/team/model-contract.ts#L239-L263](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/model-contract.ts#L239-L263)). 같은 "worker"라도 모델/CLI 계열별로 sandbox, prompt mode, 출력 파서가 달라지는 부분이다.

**문서화된 compatibility matrix는 사용 규율을 모델별로 정리한다.** `docs/agents/model-compatibility.md`는 Prometheus/Hyperplan planning, Sisyphus/Hephaestus implementation, Oracle/Aletheia review, Hermes coordination에 대해 premium·cost-effective·avoid 조합을 표로 둔다([docs/agents/model-compatibility.md#L10-L20](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/docs/agents/model-compatibility.md#L10-L20)). 설계 규칙은 planning/review는 비싼 모델, implementation은 비용 효율 슬롯, Hephaestus와 GPT-family 회피, DeepSeek 계열 budget option을 명시한다([docs/agents/model-compatibility.md#L22-L42](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/docs/agents/model-compatibility.md#L22-L42)). 이 표는 benchmark report가 아니라 usage matrix라고 스스로 범위를 제한한다([docs/agents/model-compatibility.md#L1-L8](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/docs/agents/model-compatibility.md#L1-L8)).

## 자가 점검 및 미포함 영역

**전 주장 인용 자가 점검 완료** — 본문 모든 설계 주장의 permalink는 고정 체크아웃(`deee3a446dadc9bfea31cdc8b19b00b16718082e`)의 경로·라인을 기준으로 작성했다. 단, 생성 산출물인 `bridge/*.cjs` 내부 구현과 배포 사이트는 파일 목록·빌드 경계만 사용하고 상세 내부 주장은 하지 않았다.

탐색했으나 다루지 않은 영역: `bridge/cli.cjs`, `runtime-cli.cjs`, `team-bridge.cjs`의 생성 코드 내부; `dist/` 빌드 산출물; `docs/REFERENCE.md`·웹사이트 문서 전체; `benchmarks/`와 prompt benchmark 실행 결과; `templates/` 기본 deliverables 세부; `scripts/permission-handler.mjs`의 Bash 승인 규칙 상세; notifications/HUD/telemetry 구현 상세; 개별 스킬 40여 개의 full workflow 본문 전수 비교; `tests/` 전체 회귀 케이스. 이 영역은 횡단 비교에서 특정 차원 근거가 필요해질 때 별도 표본으로 인용한다.
