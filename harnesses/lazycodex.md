## 개요

LazyCodex는 [omo (oh-my-openagent)](omo.md)를 OpenAI Codex CLI용으로 패키징한 배포판이다. 자기 정의가 "LazyCodex packages OmO as the Codex agent harness"이고, 유비도 명시적이다: "Think LazyVim for lazy.nvim, but for Codex" — 베이스 도구(Codex)에 의견이 들어간 설정·플러그인 묶음을 한 줄 설치로 얹는다는 LazyVim 모델의 차용이다([README.md#L158-L164](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/README.md#L158-L164)). 레포 스스로 "thin distribution layer"라 부르며, 코어 엔진은 `src/` 아래 oh-my-openagent 서브모듈로만 존재한다([README.md#L221](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/README.md#L221), [.gitmodules#L1-L3](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/.gitmodules#L1-L3)). 분석 체크아웃에서 이 서브모듈은 비초기화 상태라 코어 엔진 코드는 이 레포에 물리적으로 없다(`git submodule status` 결과 `-65715d1c…` — omo 분석 기준 커밋 `a4dba62…`와 다른 핀이라는 점도 함께 확인).

이 레포가 실제로 담는 것은 두 갈래의 배포 표면이다. 첫째, **npm 인스톨러 별칭** `lazycodex-ai`: 35줄짜리 bin 스크립트 하나가 `install` 인자를 받으면 `npx --yes --package oh-my-openagent omo install --platform=codex`로 셸아웃한다([bin/lazycodex-ai.js#L8-L19](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/bin/lazycodex-ai.js#L8-L19), [README.md#L49](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/README.md#L49)). 패키지가 출하하는 파일도 `bin`, `README.md`, `LICENSE` 셋뿐이다([package.json#L2-L4](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/package.json#L2-L4), [#L9-L13](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/package.json#L9-L13)). 둘째, **Codex 마켓플레이스 번들** `plugins/omo/`: `.agents/plugins/marketplace.json`이 "sisyphuslabs" 마켓플레이스의 플러그인으로 `./plugins/omo`를 가리키고([marketplace.json#L6-L16](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/.agents/plugins/marketplace.json#L6-L16)), 그 디렉토리에 훅·스킬·MCP·컴포넌트 전체가 빌드 산출물 포함으로 동봉되어 있다. 이 번들은 omo 레포 `packages/omo-codex/plugin`의 채록 사본으로 **추정**된다 — 의존성 `"@oh-my-opencode/shared-skills": "file:../../shared-skills"`는 lazycodex 레포 배치에서는 해소 불가능하고 omo 레포의 패키지 배치에서만 성립하는 상대 경로다([plugins/omo/package.json#L18-L20](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/package.json#L18-L20)).

따라서 이 문서의 분석 대상은 omo.md가 "Light 에디션"으로 한 줄 요약했던 것([omo.md 아키텍처](omo.md#아키텍처))의 실체, 즉 **opencode 플러그인 API 없이 Codex의 훅·스킬·에이전트 계약만으로 같은 하네스 의지를 재구성한 결과물**이다.

## 하네스 엔지니어링

framework.md 정의: [하네스 엔지니어링](../framework.md#1-하네스-엔지니어링-harness-engineering).

**훅이 in-process 함수에서 외부 프로세스로 바뀐다.** omo Ultimate가 단일 `PluginInterface` 객체의 TypeScript 훅 54종으로 베이스에 개입한다면([omo.md 아키텍처](omo.md#아키텍처)), Codex에는 그런 플러그인 API가 없다. LazyCodex의 모든 개입은 `hooks.json` 선언 — 이벤트별 `node <component>/dist/cli.js` 명령 실행이다. SessionStart·UserPromptSubmit·PreToolUse·PostToolUse·PostCompact·Stop·SubagentStop 7개 이벤트에 컴포넌트 CLI가 매핑되고([hooks.json#L3](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L3), [#L36](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L36), [#L92](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L92), [#L122](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L122), [#L157](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L157), [#L169](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L169)), 각 훅은 stdin으로 JSON 페이로드를 받아 stdout으로 JSON 결정을 돌려주는 타입 가드 파이프라인이다(예: [ultrawork/src/codex-hook.ts#L31-L37](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/src/codex-hook.ts#L31-L37)). 훅마다 `"LazyCodex(4.9.2): …"` statusMessage가 붙어 사용자에게 개입을 노출한다([hooks.json#L10](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L10)).

**도구 입력 게이트: create_goal 거부 훅.** PreToolUse가 `^create_goal$` 매처로 잡혀 있고([hooks.json#L81-L87](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L81-L87)), 모델이 `objective` 이외의 키(예: `token_budget`)를 넣어 호출하면 `permissionDecision: "deny"`로 차단하고 "Omit token_budget so the goal stays unlimited"라는 교정 지시를 돌려준다([codex-hook.ts#L37-L39](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ulw-loop/src/codex-hook.ts#L37-L39), [#L86-L98](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ulw-loop/src/codex-hook.ts#L86-L98), [#L155-L157](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ulw-loop/src/codex-hook.ts#L155-L157)). 베이스의 도구를 추가하는 게 아니라 베이스 도구의 *입력 형태*를 권한 결정으로 강제하는, Codex식 도구 표면 개입이다. 거부가 지키려는 것이 루프 예산(무제한 goal)이라는 점은 [루프 절](#루프-엔지니어링)에서 교차 참조한다.

**MCP 5종 동봉, 단 websearch는 없다.** `.mcp.json`이 ast_grep·grep_app·context7·git_bash·lsp를 선언하고, 로컬 서버 3종은 동봉된 `dist/cli.js`를 직접 실행한다([.mcp.json#L2-L33](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/.mcp.json#L2-L33)). omo Ultimate의 5종 내장 MCP([omo.md 하네스](omo.md#하네스-엔지니어링))와 구성이 겹치지만 websearch가 빠졌다(부재 근거: `.mcp.json` 전문이 위 5종으로 끝남). LSP를 자가 빌드 부트스트랩으로 보장하던 omo와 달리, 여기서는 빌드 산출물을 번들에 미리 넣는 방식이다.

**컨텍스트 공급은 rules 컴포넌트가 4개 이벤트에 걸쳐 담당한다.** SessionStart·UserPromptSubmit에서 프로젝트 규칙 로드, `^apply_patch$` PostToolUse에서 규칙 매칭, PostCompact에서 캐시 리셋([hooks.json#L8](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L8), [#L41](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L41), [#L111-L117](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L111-L117), [#L139](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L139)) — omo rules-injector의 Codex 대응이다. 반면 omo의 대표 하네스 장치인 hashline 편집은 Codex 번들에 없다(부재 근거: 컴포넌트 워크스페이스가 comment-checker·git-bash·rules·lsp·telemetry·start-work-continuation·ulw-loop·ultrawork 8종으로 전부다 — [plugins/omo/package.json#L8-L17](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/package.json#L8-L17)). 도구 정의 API가 없는 환경에서는 read 출력 후처리+전용 edit 도구 쌍을 만들 수 없으므로, 에디션 차이가 아니라 확장점 제약의 직접 결과로 읽힌다(이 인과는 추정).

**서브에이전트는 역할 TOML 6종으로.** `call_omo_agent` 같은 자체 도구 대신, Codex 네이티브 `spawn_agent`의 `agent_type` 인자에 매핑되는 역할 정의(explorer·librarian·plan·momus·metis·codex-ultrawork-reviewer)를 `~/.codex/agents/`에 설치한다([README.md#L138-L141](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/README.md#L138-L141)). 정의 파일은 ultrawork 컴포넌트에 동봉된 TOML로, 모델·노력 수준·developer_instructions를 한 장에 담는다([momus.toml#L3-L8](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/agents/momus.toml#L3-L8)). 어떤 에이전트를 노출하는가는 하네스, 모델 배정은 [모델별 튜닝](#모델별-튜닝) 참조.

**텔레메트리도 훅 표면의 일부다.** SessionStart마다 telemetry 컴포넌트 CLI가 돌며([hooks.json#L18-L20](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L18-L20)), 익명 `omo_codex_daily_active` 이벤트를 UTC 일 1회로 제한해 전송한다 — distinct ID는 `sha256("omo-codex:" + hostname)`, 수집 항목 한정 목록과 미수집 항목(프롬프트·소스·토큰 등) 목록, 환경변수 옵트아웃 4종이 번들 README에 공개되어 있다([plugins/omo/README.md#L20-L36](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/README.md#L20-L36)). 훅 1슬롯이 곧 기능 1단위라는 이 번들의 문법을 텔레메트리까지 따른다.

**도구 사용 자체를 조향하는 권고 훅.** PreToolUse가 `^Bash$` 매처로 git-bash 컴포넌트를 호출해 셸 git 사용 시 전용 git_bash MCP를 권고하고, PostCompact에서 그 리마인더 상태를 리셋한다([hooks.json#L70-L77](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L70-L77), [#L127-L131](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L127-L131)). 차단(deny)이 아니라 권고라는 점에서 create_goal 게이트보다 한 단계 약한 개입 등급이고, 컴팩션을 "리마인더가 증발하는 사건"으로 명시적으로 모델링한다는 점이 Codex 훅 설계의 공통 문법(PostCompact = 캐시·리마인더 리셋 이벤트)을 보여준다.

## 루프 엔지니어링

framework.md 정의: [루프 엔지니어링](../framework.md#2-루프-엔지니어링-loop-engineering).

**Stop 훅 기반 연속 강제(start-work-continuation).** omo todo-continuation-enforcer가 세션 idle 이벤트에서 미완료 todo를 검사했다면([omo.md 루프](omo.md#루프-엔지니어링)), Codex판은 Stop/SubagentStop 훅에서 `.omo/boulder.json`을 읽는다. 현재 세션(`codex:<session_id>` 접두)의 작업이 active/paused이고 플랜의 `## TODOs`/`## Final Verification Wave`에 미체크 박스가 남아 있으면 `decision: "block"`과 함께 다음 작업 디렉티브를 반환해 종료를 거부한다([codex-hook.ts#L6-L16](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/start-work-continuation/src/codex-hook.ts#L6-L16), [boulder-reader.ts#L5-L8](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/start-work-continuation/src/boulder-reader.ts#L5-L8), [#L60-L85](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/start-work-continuation/src/boulder-reader.ts#L60-L85), [#L124-L126](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/start-work-continuation/src/boulder-reader.ts#L124-L126)). 폭주 방지 장치도 Codex의 재료로 다시 만들었다: `stop_hook_active` 플래그면 개입하지 않고, 트랜스크립트에서 "context compacted" 등 컨텍스트 압박 마커 7종이 발견되면 연속을 포기한다([codex-hook.ts#L8-L9](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/start-work-continuation/src/codex-hook.ts#L8-L9), [#L42-L50](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/start-work-continuation/src/codex-hook.ts#L42-L50)). omo의 스킵 조건이 세션 상태 질의(복구 중·취소·백그라운드 실행)였다면, 여기서는 **트랜스크립트 파일의 문자열 검사**가 그 역할을 대신한다 — 플러그인이 세션 내부 상태에 접근할 수 없는 환경의 우회다.

**루프 상태가 메모리에서 저장소 파일로 내려온다.** omo ralph-loop이 인프로세스 컨트롤러(`startLoop`/`resumeLoop`/`cancelLoop`)였다면, Codex판 ulw-loop의 상태는 `.omo/ulw-loop/<session-id>/`의 brief·goals·ledger 파일이고 세션 식별은 환경변수(`CODEX_SESSION_ID` 등)에서 끌어온다([paths.ts#L8](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ulw-loop/src/paths.ts#L8), [#L33-L36](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ulw-loop/src/paths.ts#L33-L36)). 스킬은 이 상태를 CLI로만 조작하고 컴팩션 후에는 ledger부터 재독하라고 강제하며([ulw-loop/SKILL.md#L22-L24](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/ulw-loop/SKILL.md#L22-L24)), 사용자 프롬프트의 조향 디렉티브는 UserPromptSubmit 훅이 파싱해 같은 상태에 반영한다([codex-hook.ts#L63-L80](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ulw-loop/src/codex-hook.ts#L63-L80)). 루프 컨트롤러 하나가 "디스크 상태 + CLI + Stop 훅 + 스킬 프롬프트" 네 조각으로 분해된 형태다.

**완료 검증 사이클은 스킬 텍스트로 산다.** start-work는 체크박스마다 5게이트(플랜 재독→자동 검증→Manual-QA 실물 아티팩트→적용 가능한 ultraqa 9클래스 전수 프로브→정리 영수증)를 요구하고([start-work/SKILL.md#L117-L123](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/start-work/SKILL.md#L117-L123), [#L113](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/start-work/SKILL.md#L113)), 워커의 완료 주장을 `DoneClaim`→독립 컨텍스트의 `AdversarialVerify`→`confirmed`만 통과시키는 완료 계약으로 재검증한다([#L127-L155](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/start-work/SKILL.md#L127-L155)). omo가 "완료 주장의 구조적 불신"을 훅 코드로 구현했다면, Codex판은 같은 정책의 상당 부분을 **프롬프트로만** 들고 있고 코드 강제는 Stop 훅 하나다 — 메커니즘-내용 분리 규칙상 주입 장치(Stop 훅)는 루프, 게이트 절차 텍스트는 [프롬프트](#프롬프트-엔지니어링) 소속인데, 정책의 무게중심 자체가 프롬프트 쪽으로 이동한 것이 Codex 이식의 특징이다.

**서브에이전트 수명 주기 규율도 프롬프트화됐다.** omo의 스폰 깊이 계산·강제 백그라운드 같은 코드 규칙 대신, "spawn은 background로, 짧은 `wait_agent` 사이클로 폴링, 타임아웃은 실패가 아님, `WORKING:`/`BLOCKED:` 신호 규약, 미달 시 더 작은 태스크로 재스폰"이 스킬 본문의 비협상 조항으로 들어가 있다([ulw-loop/SKILL.md#L28-L33](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/ulw-loop/SKILL.md#L28-L33)).

**반복 상한은 모드별 이원화.** README의 커맨드 계약상 `$ulw-loop`는 Oracle 검증 완료까지 도는 자기 참조 루프이고, 반복 상한이 ultrawork 모드 500회/일반 모드 100회로 갈린다([README.md#L83](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/README.md#L83)). 진입 경로의 루프 연결도 있다: `$start-work`가 선택 가능한 플랜 없이 호출되면 그 발화 자체를 승인으로 간주해 ulw-plan을 부트스트랩하고, 플랜이 생기면 곧장 실행 페이즈로 넘어간다 — 멈춰서 되묻는 대신 계획 루프를 실행 루프 앞에 자동 삽입하는 설계다([start-work/SKILL.md#L57-L65](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/start-work/SKILL.md#L57-L65)).

## 프롬프트 엔지니어링

framework.md 정의: [프롬프트 엔지니어링](../framework.md#3-프롬프트-엔지니어링-prompt-engineering).

**스킬 이식이 손번역이 아니라 기계 변환이다.** 빌드 단계의 `sync-skills.mjs`가 공유 스킬을 omo에서 복사한 뒤, opencode 전용 오케스트레이션 호출(`call_omo_agent`/`task`/`background_output`/`team_*`)을 정규식으로 감지하면([sync-skills.mjs#L19](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/sync-skills.mjs#L19)) 스킬 머리에 "Codex Harness Tool Compatibility" 절을 주입한다 — OpenCode 도구 호출 예시를 `multi_agent_v1.spawn_agent` 호출로 옮기는 대응표와 "아래 코드 블록이 이 절과 충돌하면 이 절이 이긴다"는 우선순위 선언이다([sync-skills.mjs#L21-L39](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/sync-skills.mjs#L21-L39), [#L80-L100](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/sync-skills.mjs#L80-L100)). 본문을 고치지 않고 해석 규칙을 앞에 박는 전략이라, 산출물에는 OpenCode 시절의 `task(...)` 예시가 그대로 남은 채 번역 지시만 추가된 스킬이 존재한다([init-deep/SKILL.md#L5-L21](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/init-deep/SKILL.md#L5-L21) — 본문 Phase 1은 여전히 `task(subagent_type="explore", …)` 형태다, [#L70-L75](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/init-deep/SKILL.md#L70-L75)).

**스킬 본문은 2단 로딩으로 압축된다.** ulw-loop·ulw-plan 스킬은 의도적으로 컴팩트한 SKILL.md와 `references/full-workflow.md` 분리 구조를 쓰고, "현재 페이즈에 필요한 절만 읽고 정확히 실행하라"고 지시한다([ulw-loop/SKILL.md#L12](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/ulw-loop/SKILL.md#L12), [ulw-plan/SKILL.md#L12](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/ulw-plan/SKILL.md#L12)) — Codex 컴포저에서 `$` 접두로 호출되는 스킬 디스크립션 예산([README.md#L62-L65](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/README.md#L62-L65), [#L99-L100](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/README.md#L99-L100))을 아끼는 점진적 공개다.

**스킬별 오버레이는 문자열 치환으로 깊어진다.** 같은 스크립트가 start-work의 완료 절차 원문을 Codex판으로 통째로 치환해 "Global Review and Debugging Gate"(review-work 5레인 전부 PASS + 실패 가설 3개 이상 런타임 감사 + 증거 위생)를 삽입하고([sync-skills.mjs#L109-L124](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/sync-skills.mjs#L109-L124), 결과물: [start-work/SKILL.md#L171-L176](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/start-work/SKILL.md#L171-L176)), review-work에는 "타임아웃·ack-only·결론 불능 레인은 통과가 아니라 실패"라는 블로킹 게이트 절을 추가한다([sync-skills.mjs#L126-L141](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/sync-skills.mjs#L126-L141)). 프롬프트 내용의 분기 기준이 모델이 아니라 **타깃 하네스**라는 점이 omo의 모델별 프롬프트 변형과 대비된다.

**키워드 트리거는 축소 이식.** `\b(?:ultrawork|ulw)\b` 정규식 하나가 UserPromptSubmit에서 ultrawork 디렉티브를 `additionalContext`로 주입한다([ultrawork/src/codex-hook.ts#L5](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/src/codex-hook.ts#L5), [#L31-L37](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/src/codex-hook.ts#L31-L37)). 재주입 중복 방지는 omo가 시스템 프롬프트 태그 검사로 했던 것을, 트랜스크립트 꼬리 512KB에서 `<ultrawork-mode>` 마커를 찾는 방식으로 옮겼다([#L6-L7](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/src/codex-hook.ts#L6-L7), [#L39-L75](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/src/codex-hook.ts#L39-L75)).

**주입 디렉티브의 단일 소스는 마크다운 파일이다.** 훅이 주입하는 텍스트는 코드에 박힌 문자열이 아니라 컴포넌트 루트의 `directive.md`이고, TypeScript는 그 파일을 읽어 export하는 두 줄짜리 로더다([ultrawork/src/directive.ts#L1-L3](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/src/directive.ts#L1-L3)). 361줄짜리 ultrawork 디렉티브는 `ULTRAWORK MODE ENABLED!` 첫 줄 강제, "TESTS ALONE NEVER PROVE DONE", LIGHT/HEAVY 티어 분류 같은 행동 계약을 담는다([ultrawork/directive.md#L1-L31](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/directive.md#L1-L31)). 코드 변경 없이 이 파일만 고쳐도 행동이 바뀌는, framework.md 판별 질문 그대로의 프롬프트 산출물이다.

**플래너 페르소나는 유지된다.** ulw-plan은 여전히 Prometheus다: 탐색으로 답할 수 있는 질문을 사용자에게 묻는 것을 실패로 규정하고, 명시적 승인 전 플랜 생성을 금지하며, 산출물은 `.omo/plans/<slug>.md`로 한정한다([ulw-plan/SKILL.md#L10](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/ulw-plan/SKILL.md#L10), [#L20-L25](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/ulw-plan/SKILL.md#L20-L25)). 에이전트 역할 TOML의 developer_instructions도 프롬프트 산출물이다 — 리뷰어는 "UNCONDITIONAL APPROVAL 아니면 REJECTION, 'looks good but…'은 거부로 취급"이라는 이진 판정 규칙을 받는다([codex-ultrawork-reviewer.toml#L6-L19](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/agents/codex-ultrawork-reviewer.toml#L6-L19)).

## 아키텍처

framework.md 정의: [아키텍처](../framework.md#4-아키텍처-architecture).

**배포 형태 자체가 이 레포의 본질이다.** 제품 코드가 사는 곳(omo 레포)과 배포 정체성이 사는 곳(이 레포)을 분리하고, 후자에는 ① npm 별칭 bin 한 장, ② 마켓플레이스 번들 `plugins/omo/`, ③ 마케팅·문서 사이트 `packages/web/`(Next.js, lazycodex.ai), ④ 코어 서브모듈 핀만 둔다([README.md#L223-L232](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/README.md#L223-L232)). 인스톨러 별칭의 정체성은 코드가 아니라 **이름과 기본 플래그**다 — `lazycodex-ai install`은 `omo install --platform=codex`의 고정 표기일 뿐이다([bin/lazycodex-ai.js#L8-L19](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/bin/lazycodex-ai.js#L8-L19)).

**소스 오브 트루스 경계를 CI가 집행한다.** 이 레포로 들어오는 모든 PR은 `pull_request_target` 워크플로가 자동으로 닫으면서 "변경은 oh-my-openagent의 packages/omo-codex로 보내라"는 안내와 코딩 에이전트용 재제출 프롬프트까지 코멘트로 남긴다([pr-source-guidance.yml#L4-L5](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/.github/workflows/pr-source-guidance.yml#L4-L5), [#L26-L35](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/.github/workflows/pr-source-guidance.yml#L26-L35)). omo가 패키지 경계를 결합도 감사 테스트로 강제했다면([omo.md 아키텍처](omo.md#아키텍처)), 여기서는 레포 경계를 CI로 강제한다. 같은 결로, 공개 표면의 카피도 테스트로 고정된다: 추적 파일 전체에 한글 금지(`plugins/omo/` 제외), 구 패키지 러너 표기 금지, 런치 게이팅 문구 금지([no-korean-text.test.mjs#L6-L7](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/test/no-korean-text.test.mjs#L6-L7), [no-bunx-text.test.mjs#L6-L11](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/test/no-bunx-text.test.mjs#L6-L11), [no-launch-gating-text.test.mjs#L6-L14](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/test/no-launch-gating-text.test.mjs#L6-L14)).

**별칭의 정체성도 테스트로 고정된다.** 인스톨러 bin은 `--dry-run` 모드가 실제 실행할 npx 명령 문자열을 출력하게 되어 있고([bin/lazycodex-ai.js#L21-L24](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/bin/lazycodex-ai.js#L21-L24)), 루트 테스트가 그 출력이 정확히 `npx --yes --package oh-my-openagent omo install --platform=codex …`인지, 패키지명·버전·퍼블리시 워크플로 기본값이 릴리스 버전과 일치하는지를 단언한다([lazycodex-ai-bin.test.mjs#L47-L63](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/test/lazycodex-ai-bin.test.mjs#L47-L63), [#L29-L45](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/test/lazycodex-ai-bin.test.mjs#L29-L45)). 제품이 "위임 명령 한 줄"일 때 그 한 줄이 곧 회귀 대상이 된다.

**번들은 "원자 컴포넌트 8종 + 어그리게이트 매니페스트" 구조다.** 각 컴포넌트(rules, lsp, comment-checker, git-bash, ultrawork, ulw-loop, start-work-continuation, telemetry)는 자체 hooks/skills를 가진 독립 워크스페이스로 유지하되, 루트 매니페스트 하나가 `omo`라는 단일 플러그인 네임스페이스로 skills·hooks·mcpServers를 합산 노출한다([plugins/omo/README.md#L3-L16](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/README.md#L3-L16), [plugin.json#L21-L23](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/.codex-plugin/plugin.json#L21-L23)). omo의 54종 훅 enum과 같은 "항목 원자화" 구조 결정이 Codex에서는 빌드 스크립트 체인(sync-version→sync-hook-status-messages→sync-skills→build-components)으로 실현된다([plugins/omo/package.json#L22](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/package.json#L22)). 브랜딩도 이 체인의 산출물이다: 훅 상태 메시지에서 "OMO" 단어를 제거하고 `LazyCodex(<version>):` 접두로 정규화하는 변환기가 따로 있다([hook-status-message.mjs#L1](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/hook-status-message.mjs#L1), [#L8-L15](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/hook-status-message.mjs#L8-L15)).

**번들은 스토어 등재를 전제한 아티팩트다.** 플러그인 매니페스트가 기능 목록(capabilities)·권장 첫 프롬프트(defaultPrompt)·브랜드 컬러까지 담은 인터페이스 메타데이터를 갖고([plugin.json#L30-L46](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/.codex-plugin/plugin.json#L30-L46)), 마켓플레이스 항목은 설치 정책 `AVAILABLE`/인증 시점 `ON_INSTALL`을 선언한다([marketplace.json#L11-L14](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/.agents/plugins/marketplace.json#L11-L14)). npm 인스톨러와 마켓플레이스 번들이라는 두 설치 경로가 한 레포에 공존하는 셈인데, 두 경로가 어떻게 동기화되는지는 이 레포만으로는 보이지 않는다(추정: upstream 빌드가 양쪽에 산출물을 밀어 넣는다).

**하네스가 스스로를 재설치하고, 설정을 마이그레이션한다.** SessionStart(`^startup$`)마다 auto-update 훅이 돌고([hooks.json#L25-L31](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L25-L31)), 24시간 스로틀을 지나면 `npx --yes lazycodex-ai@latest install --no-tui --codex-autonomous`를 실행한다([auto-update.mjs#L19-L23](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/auto-update.mjs#L19-L23)). 같은 경로의 설정 마이그레이션은 `~/.codex/config.toml`의 관리 항목을 모델 카탈로그 기준으로 갱신하되, 사용자가 직접 바꾼 값은 "user-modified"로 분류해 영구히 비관리로 둔다([migrate-codex-config.mjs#L69-L80](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/migrate-codex-config.mjs#L69-L80)).

**베이스 플랫폼 버그에 대한 런타임 방어가 코드로 박혀 있다.** 마이그레이션은 매 세션 시작마다 Codex의 `multi_agent_v2` 기능을 `enabled = false`로 강제 재기록한다 — 근거 주석이 upstream 이슈(openai/codex#26753: 플래그를 켜면 모든 턴이 HTTP 400)와 옵트아웃 환경변수까지 명시한다([multi-agent-v2-guard.mjs#L11-L24](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/migrate-codex-config/multi-agent-v2-guard.mjs#L11-L24), [#L26-L28](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/migrate-codex-config/multi-agent-v2-guard.mjs#L26-L28)). 한편 README와 스킬 호환 절은 "multi_agent_v2 세션에서 installer가 agent_type을 노출한다"고 서술한다([README.md#L147-L150](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/README.md#L147-L150)). 두 표면이 공존하는 이유(V2가 고쳐질 때를 대비한 잔존 서술로 보인다)는 추정이다.

**설치 건강을 스킬로 진단한다.** lcx-doctor는 최신 lazycodex·openai/codex 소스를 `/tmp`에 채록해 로컬 설치본과 비교하고, 발견된 FAIL을 GitHub 이슈 검색으로 기지(旣知) 여부까지 확인하는 읽기 전용 진단 스킬이다([lcx-doctor/SKILL.md#L10](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/lcx-doctor/SKILL.md#L10), [#L16-L31](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/lcx-doctor/SKILL.md#L16-L31)) — 배포판이라는 정체성이 진단 표면(`lcx-*` 스킬군)까지 만들어낸 사례다.

## 모델별 튜닝

framework.md 정의: [모델별 튜닝](../framework.md#5-모델별-튜닝-per-model-tuning).

**라우팅의 축이 멀티 프로바이더에서 단일 프로바이더 내 등급 분기로 좁아진다.** omo는 모델사별 카테고리 합성과 에이전트별 7단 폴백 체인을 가졌지만([omo.md 모델별 튜닝](omo.md#모델별-튜닝)), Codex 번들의 역할 TOML은 전부 OpenAI 모델이다: 탐색·사서는 `gpt-5.4-mini` + `model_reasoning_effort = "low"` + `service_tier = "fast"`, 플래너와 플랜 리뷰어(momus)는 `gpt-5.5` + `xhigh`, 사전 분석(metis)과 검증 리뷰어는 `gpt-5.5` + `high`다([explorer.toml#L4-L6](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/agents/explorer.toml#L4-L6), [plan.toml#L4-L5](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/agents/plan.toml#L4-L5), [momus.toml#L4-L5](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/agents/momus.toml#L4-L5), [codex-ultrawork-reviewer.toml#L4-L5](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/agents/codex-ultrawork-reviewer.toml#L4-L5)). 리뷰어 지시문은 기본 모델 선택의 이유까지 명시한다: "ChatGPT 계정 호환 프런티어 모델을 의도적으로 쓴다"([codex-ultrawork-reviewer.toml#L10](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/agents/codex-ultrawork-reviewer.toml#L10)). README는 이 분기를 쿼터 규율로 정당화한다 — quick은 `gpt-5.4-mini`, ultrabrain은 고추론 GPT, 벤치마크 근거 링크 동반([README.md#L185-L196](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/README.md#L185-L196)).

**모델 설정이 버전드 카탈로그로 배포된다.** `model-catalog.json`이 현재 권장 프로필(gpt-5.5, 컨텍스트 윈도 400k, effort high, 플랜 모드 xhigh)과 역할별(default/verifier/worker) 프로필을 선언하고([model-catalog.json#L2-L24](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/model-catalog.json#L2-L24)), `managedProfiles`에 과거 배포 프로필(1M·272k 윈도)을 매칭 패턴으로 열거해 둔다([#L25-L42](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/model-catalog.json#L25-L42)). 마이그레이션이 사용자의 현재 config.toml 값을 이 레거시 프로필과 대조해 "우리가 과거에 써준 값"일 때만 새 권장값으로 덮어쓴다([migrate-codex-config.mjs#L69-L80](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/migrate-codex-config.mjs#L69-L80)) — 모델 분기 데이터(카탈로그)는 이 차원, 그것을 적용하는 관리 구역 메커니즘은 [아키텍처](#아키텍처) 소속이다.

**스킬 산문이 특정 모델의 문체를 지목한다.** lcx-doctor는 본문에서 "Use GPT-5.5 style: outcome first, concise, evidence-bound"라고 출력 스타일의 준거 모델을 명시한다([lcx-doctor/SKILL.md#L12](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/lcx-doctor/SKILL.md#L12)). omo가 모델별 프롬프트 *파일*을 분기했다면, 여기서는 단일 프롬프트 안에서 기대 모델의 문체를 호명하는 가장 가벼운 형태의 모델별 튜닝이다.

**부재 확인:** omo식 에이전트-모델 결합 강제(no-sisyphus-gpt류 런타임 스왑)나 모델별 프롬프트 변형 파일은 이 번들에 없다(탐색 경로: `plugins/omo/` 전체에서 `sisyphus`·`claude`·`gemini` 모델 분기 패턴 검색 — 매치는 README·스킬의 서사적 언급뿐, 코드 분기 없음; 컴포넌트 8종 워크스페이스 전수 확인). 타깃 플랫폼이 단일 모델 계열이므로 모델 판별 분기 자체가 소거된 것으로 추정된다.
