# Ouroboros (ooo) — 심층 분석

- **정식 레포명:** Q00/ouroboros (약칭: ouroboros, ooo)
- **분석 기준 커밋:** `b032f3a96fa2a5388affd55a2042a5a439fbe5f3`
- **클론 채록일:** 2026-06-12 / **분석일:** 2026-06-12
- **분석 축:** 스펙 주도 Agent OS. 플러그인형 harness 자체보다 상위의 Seed·Ledger·Runtime 계약을 중심에 두고, `interview → seed → execute → evaluate → evolve`가 하네스·루프·프롬프트·아키텍처·모델 분기로 나뉘는 방식을 분석한다.

## 개요

Ouroboros는 첫 화면에서 “Stop prompting. Start specifying.”와 “Agent OS for replayable, specification-first AI coding workflows”를 전면에 둔다([README.md#L18-L22](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/README.md#L18-L22)). 같은 README는 vague idea를 Claude Code, Codex CLI, OpenCode, Hermes, Gemini, Kiro, Copilot, Pi 전반에서 verified working codebase로 바꾸는 도구라고 설명하고, local-first runtime layer가 non-deterministic agent work를 replayable·observable·policy-bound execution contract로 바꾼다고 정의한다([README.md#L39-L45](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/README.md#L39-L45)).

README의 Agent OS stack은 shell `Q00/ourocode`, apps `Q00/ouroboros-plugins`, OS `Q00/ouroboros` 세 레이어로 나뉘며, 이 레포는 Seed·Ledger·Runtime·MCP·safety boundaries를 소유하는 core/kernel로 놓인다([README.md#L49-L60](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/README.md#L49-L60)). 연결 설명도 kernel이 contract를 소유하고 모든 action을 Seed-bound, ledger-recorded, replayable event로 만든다고 적으며, plugin은 scoped capability를 선언하고, ourocode는 MCP state·interview question·wonderTool decision을 TUI에 노출한다고 적는다([README.md#L61-L79](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/README.md#L61-L79)).

핵심 문제 정의는 “input clarity”다. README는 vague prompt가 AI guessing과 rework를 만들고, Ouroboros가 Socratic interview, immutable seed spec, 3-stage automated evaluation gate로 이를 다룬다고 비교한다([README.md#L89-L99](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/README.md#L89-L99), [README.md#L219-L228](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/README.md#L219-L228)). 따라서 이 문서에서 Ouroboros는 “다른 에이전트에 붙는 플러그인 묶음”보다 “스펙을 실행 계약으로 승격시키는 workflow OS”로 다룬다.

## 하네스 엔지니어링

framework.md 정의: [하네스 엔지니어링](../framework.md#1-하네스-엔지니어링-harness-engineering).

**명령 표면은 `ooo` skill과 `ouroboros` CLI가 함께 만든다.** AGENTS.md는 `ooo auto`, `ooo interview`, `ooo seed`, `ooo run`, `ooo evaluate`, `ooo evolve`, `ooo status`, `ooo ralph`, `ooo pm`, `ooo brownfield`, `ooo publish`, `ooo resume-session` 같은 입력을 각각의 `skills/*/SKILL.md`로 라우팅한다고 적는다([AGENTS.md#L10-L38](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/AGENTS.md#L10-L38)). CLI 쪽에서는 `src/ouroboros/cli/main.py`가 Typer app을 만들고 `auto`, `init`, `run`, `job`, `config`, `status`, `cancel`, `mcp`, `setup`, `detect`, `tui`, `pm`, `qa`, `plugin`, `resume`, `uninstall`, `workflow-ir`를 등록한다([src/ouroboros/cli/main.py#L74-L112](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/cli/main.py#L74-L112)). 등록되지 않은 top-level command는 `_PluginAwareGroup`이 plugin dispatch를 시도하도록 되어 있어, 하네스의 명령 공간은 first-party command와 installed UserLevel plugin을 같은 CLI group 아래에 둔다([src/ouroboros/cli/main.py#L49-L72](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/cli/main.py#L49-L72)).

**Claude plugin 포장은 skill·MCP 서버를 사용자 런타임에 싣는 배포용 하네스 표면이다.** `.claude-plugin/plugin.json`은 plugin name, version, description, repository, keywords, `skills: ./skills/`, `mcpServers: ./.mcp.json`를 선언한다([.claude-plugin/plugin.json#L1-L24](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/.claude-plugin/plugin.json#L1-L24)). `.claude-plugin/.mcp.json`은 `uvx --from ouroboros-ai[mcp,claude] ouroboros mcp serve`를 MCP server command로 등록한다([.claude-plugin/.mcp.json#L1-L9](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/.claude-plugin/.mcp.json#L1-L9)). AGENTS.md도 end user는 Codex plugin marketplace로 설치하면 skills/hooks/agents가 AGENTS 파일 없이 native로 동작한다고 설명한다([AGENTS.md#L1-L8](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/AGENTS.md#L1-L8)).

**Seed는 모델에게 넘길 작업 문맥을 정적 계약으로 만드는 핵심 context substrate다.** `core/seed.py`는 Seed를 workflow의 constitution이라고 부르며, ambiguity 값이 0.2 이하일 때 Big Bang interview에서 생성된 immutable specification이라고 설명한다([src/ouroboros/core/seed.py#L1-L15](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/core/seed.py#L1-L15)). Seed의 direction인 goal·constraints·acceptance_criteria는 workflow가 달성할 WHAT을 정의하고 생성 후 변경할 수 없으며 evaluation의 ground truth가 된다고 명시한다([src/ouroboros/core/seed.py#L182-L199](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/core/seed.py#L182-L199)). 같은 모델은 brownfield context, ontology schema, evaluation principles, exit conditions, metadata를 필드로 담아 실행·검증에 필요한 정보를 한 객체에 묶는다([src/ouroboros/core/seed.py#L238-L279](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/core/seed.py#L238-L279)).

**Ledger/EventStore는 모델 호출 바깥의 세계 상태를 replay 가능한 형태로 만든다.** EventStore는 SQLAlchemy Core와 aiosqlite backend로 append/replay/close를 제공하는 async event store이며 모든 operation을 transactional로 다룬다고 주석과 docstring이 설명한다([src/ouroboros/persistence/event_store.py#L1-L5](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/persistence/event_store.py#L1-L5), [src/ouroboros/persistence/event_store.py#L101-L119](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/persistence/event_store.py#L101-L119)). 기본 저장 위치는 `~/.ouroboros/ouroboros.db`이고, read-only mode에서는 SQLite URL을 `mode=ro&uri=true` 형태로 바꾸어 accidental write path가 실패하게 한다([src/ouroboros/persistence/event_store.py#L121-L149](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/persistence/event_store.py#L121-L149), [src/ouroboros/persistence/event_store.py#L151-L177](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/persistence/event_store.py#L151-L177)). raw subscribed runtime stream payload는 BaseEvent record로 normalize하기 전 persistence를 거부하므로, ledger는 “무엇이든 저장”이 아니라 이벤트 형태를 하네스가 강제하는 장치다([src/ouroboros/persistence/event_store.py#L179-L205](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/persistence/event_store.py#L179-L205)).

**Workflow IR는 agent/plugin/verifier work를 dispatch하기 전 typed node와 evidence contract로 감싼다.** `workflow_ir.py`는 WorkflowSpec, WorkflowNode, WorkflowEdge, validate_workflow를 정의하고, dangling edge·duplicate id·unreachable terminal·missing evidence/output schema를 deterministic validator가 거부한다고 적는다([src/ouroboros/orchestrator/workflow_ir.py#L1-L27](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/orchestrator/workflow_ir.py#L1-L27)). `WorkflowNode`는 owner를 harness/agent/plugin/human_gate/verifier로 구분하고 capability_envelope, runtime_hints, metadata를 담으며, agent/plugin/verifier owner에는 evidence_schema_ref를 요구한다([src/ouroboros/orchestrator/workflow_ir.py#L194-L281](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/orchestrator/workflow_ir.py#L194-L281), [src/ouroboros/orchestrator/workflow_ir.py#L283-L309](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/orchestrator/workflow_ir.py#L283-L309)).

**Plugin manifest와 trust store는 외부 capability를 하네스 계약으로 제한한다.** `plugin/manifest.py`는 UserLevel plugin manifest를 schema version별 vendored JSON Schema로 검증하고, schema violation을 path/json_pointer/expected/got이 있는 `PluginManifestError`로 표면화한다고 설명한다([src/ouroboros/plugin/manifest.py#L1-L23](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/plugin/manifest.py#L1-L23)). manifest data model에는 command spec, capability, permission, source, entrypoint, hook, audit event, action descriptor가 분리되어 있다([src/ouroboros/plugin/manifest.py#L118-L184](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/plugin/manifest.py#L118-L184), [src/ouroboros/plugin/manifest.py#L230-L263](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/plugin/manifest.py#L230-L263)). `trust_store.py`는 trust metadata를 plugin install root 밖 `~/.ouroboros/trust`에 두어 artifact tree hash가 trust file 때문에 변하지 않게 하고, scope grant는 exact string으로만 검사하며 parent scope가 child를 암시하지 않는다고 적는다([src/ouroboros/plugin/trust_store.py#L36-L47](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/plugin/trust_store.py#L36-L47), [src/ouroboros/plugin/trust_store.py#L72-L108](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/plugin/trust_store.py#L72-L108)).

## 루프 엔지니어링

framework.md 정의: [루프 엔지니어링](../framework.md#2-루프-엔지니어링-loop-engineering).

**첫 루프는 ambiguity gate다.** `bigbang/ambiguity.py`는 requirement clarity를 측정하고 ambiguity 값이 0.2 이하일 때 Seed generation으로 진행한다고 설명하며, goal clarity 40%, constraint clarity 30%, success criteria clarity 30%로 가중치를 둔다([src/ouroboros/bigbang/ambiguity.py#L1-L10](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/ambiguity.py#L1-L10)). 같은 파일은 threshold 0.2, seed closer threshold 0.25, auto-complete streak 2, per-dimension floor, brownfield weights, scoring temperature 0.1을 상수로 둔다([src/ouroboros/bigbang/ambiguity.py#L34-L57](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/ambiguity.py#L34-L57)). `AmbiguityScore.is_ready_for_seed`도 overall score가 threshold 이하인지 검사하는 property로 구현되어 있다([src/ouroboros/bigbang/ambiguity.py#L188-L208](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/ambiguity.py#L188-L208)).

**Interview loop는 prompt-budget safety와 사용자 제어를 같이 둔다.** `interview.py`는 vague idea를 iterative questioning으로 clear requirements로 다듬는 engine이며 사용자가 언제 멈출지 제어한다고 설명한다([src/ouroboros/bigbang/interview.py#L1-L5](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/interview.py#L1-L5)). constants는 early exit 전 최소 답변 round, default reference round, prompt-safe initial context limit, Agent SDK CLI empty response empirical boundary, safe prompt budget을 둔다([src/ouroboros/bigbang/interview.py#L32-L70](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/interview.py#L32-L70)). toolless interview base prompt는 exactly one Socratic question, no file/API/tool exploration, biggest unresolved decision targeting을 요구한다([src/ouroboros/bigbang/interview.py#L71-L87](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/interview.py#L71-L87)).

**Interview state는 resume와 scoring state를 함께 가진다.** `InterviewState`는 interview_id, rounds, initial_context, is_brownfield, codebase_paths, codebase_context, ambiguity_score, ambiguity_breakdown, completion_candidate_streak, lateral_review_advised_milestones를 갖는다([src/ouroboros/bigbang/interview.py#L162-L192](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/interview.py#L162-L192)). `start_interview`는 cwd가 있으면 brownfield를 auto-detect하고, first-question failure 뒤에도 resumable handle이 남도록 freshly-created state를 즉시 저장한다([src/ouroboros/bigbang/interview.py#L354-L422](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/interview.py#L354-L422)). `ask_next_question`은 history와 system prompt를 serialized CLI budget 안에 맞춰 trim하고, clarification role로 LLM adapter를 호출한다([src/ouroboros/bigbang/interview.py#L424-L523](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/interview.py#L424-L523)).

**Execution loop는 Double Diamond phase machine이다.** `double_diamond.py`는 Discover/Define/Design/Deliver 네 단계를 diverge/converge 패턴으로 설명하고, phase transition event와 retry logic을 포함한다고 적는다([src/ouroboros/execution/double_diamond.py#L1-L16](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/execution/double_diamond.py#L1-L16)). 같은 파일은 AC child dependency를 Kahn algorithm으로 level grouping하여 같은 level을 parallel execution 가능 단위로 묶고, cycle detection 시 sequential fallback을 둔다([src/ouroboros/execution/double_diamond.py#L56-L144](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/execution/double_diamond.py#L56-L144)). Phase enum은 Discover/Design을 divergent, Define/Deliver를 convergent로 분류한다([src/ouroboros/execution/double_diamond.py#L239-L263](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/execution/double_diamond.py#L239-L263)).

**Evaluation loop는 mechanical → semantic → consensus 순서로 멈춤 조건을 둔다.** `evaluation/pipeline.py`는 Stage 1 Mechanical Verification, Stage 2 Semantic Evaluation, Stage 3 Multi-Model Consensus를 orchestrate한다고 설명한다([src/ouroboros/evaluation/pipeline.py#L1-L9](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/evaluation/pipeline.py#L1-L9)). `evaluate`는 Stage 1 lint/build/test/static/coverage가 실패하면 중단하고, Stage 2 AC compliance가 실패하면 consensus override가 없을 때 중단하며, trigger가 켜지면 Stage 3 consensus로 final approval을 산출한다([src/ouroboros/evaluation/pipeline.py#L91-L164](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/evaluation/pipeline.py#L91-L164), [src/ouroboros/evaluation/pipeline.py#L166-L252](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/evaluation/pipeline.py#L166-L252)).

**Evolution loop는 generation 단위로 output을 다시 seed로 먹인다.** README의 loop diagram은 Interview → Seed → Execute → Evaluate에서 output of evaluation이 next generation seed specification으로 들어간다고 설명하며, convergence를 ontology similarity 0.95 이상으로 둔다([README.md#L232-L257](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/README.md#L232-L257)). `evolution/loop.py`도 Gen 1은 Seed(O₁) → Execute → Validate → Evaluate, Gen 2+는 Wonder → Reflect → Seed(O₂) → Execute → Validate → Evaluate로 닫힌 cycle을 설명한다([src/ouroboros/evolution/loop.py#L1-L12](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/evolution/loop.py#L1-L12)). Config는 max_generations 30, convergence_threshold 0.95, stagnation_window 3, min_generations 3, eval gate controls를 가진다([src/ouroboros/evolution/loop.py#L80-L93](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/evolution/loop.py#L80-L93)).

**Resilience loop는 반복 실패의 모양을 감지한다.** `resilience/stagnation.py`는 spinning, oscillation, no drift, diminishing returns 네 stagnation pattern을 감지한다고 설명하고, hash-based comparison과 event emission을 설계로 둔다([src/ouroboros/resilience/stagnation.py#L1-L15](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/resilience/stagnation.py#L1-L15)). `StagnationDetector.detect`는 네 detector를 모두 실행하고 감지된 pattern을 log event로 남긴다([src/ouroboros/resilience/stagnation.py#L153-L245](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/resilience/stagnation.py#L153-L245)).

## 프롬프트 엔지니어링

framework.md 정의: [프롬프트 엔지니어링](../framework.md#3-프롬프트-엔지니어링-prompt-engineering).

**Skill Markdown은 user-facing `ooo` command의 자연어 실행 계약이다.** AGENTS.md는 command 입력마다 대응하는 `skills/<name>/SKILL.md`를 읽고 따르라고 명령하며, Skill tool을 쓰지 말고 파일을 직접 읽으라고 적는다([AGENTS.md#L10-L38](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/AGENTS.md#L10-L38)). `skills/interview/SKILL.md`는 Required Skill Capabilities, Non-Skippable Gates, Version Check, MCP tool discovery, Path A/B decision, code-answer routing, lateral review dispatch를 긴 절차문으로 둔다([skills/interview/SKILL.md#L15-L36](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/skills/interview/SKILL.md#L15-L36), [skills/interview/SKILL.md#L88-L152](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/skills/interview/SKILL.md#L88-L152)). 이 Markdown들은 코드가 아니라 모델이 따라야 할 user-session playbook이라는 점에서 프롬프트 엔지니어링으로 분류된다.

**Socratic interview의 핵심 지시문은 “질문만 생성”하는 역할 경계를 반복한다.** `agents/socratic-interviewer.md`는 expert requirements engineer라고 정의하면서도 “ONLY an interviewer”, “NEVER promise to build demos, write code, or execute anything”라고 금지한다([src/ouroboros/agents/socratic-interviewer.md#L1-L10](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/agents/socratic-interviewer.md#L1-L10)). 같은 prompt는 question generator가 caller-provided existing-system context만 쓰고, brownfield에서는 `[from-code]` factual answer와 `[from-user]` decision을 구분하며, biggest ambiguity와 ontological questions를 target하라고 요구한다([src/ouroboros/agents/socratic-interviewer.md#L11-L38](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/agents/socratic-interviewer.md#L11-L38)). `interview.py`의 internal base prompt도 exactly one Socratic question, no tools, no file exploration, biggest unresolved decision targeting을 코드 문자열로 보관한다([src/ouroboros/bigbang/interview.py#L71-L87](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/interview.py#L71-L87)).

**Persona prompts는 loop 중 perspective switch를 텍스트로 제공한다.** `InterviewPerspective`는 researcher, simplifier, architect, breadth-keeper, seed-closer를 enum으로 두고, `_load_interview_perspective_strategies`가 `src/ouroboros/agents/{name}.md`에서 persona prompt data를 lazy-load한다([src/ouroboros/bigbang/interview.py#L90-L136](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/interview.py#L90-L136)). Ontologist prompt는 Essence, Root Cause, Prerequisites, Hidden Assumptions 네 질문을 “What IS this, really?” 같은 문장으로 둔다([src/ouroboros/agents/ontologist.md#L1-L27](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/agents/ontologist.md#L1-L27)). Architect prompt도 structural symptom, current structure, root misalignment, restructuring을 질문 세트와 output shape로 준다([src/ouroboros/agents/architect.md#L1-L62](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/agents/architect.md#L1-L62)).

**Seed·Evaluator prompts는 스펙 생성과 검증 판단을 형식화한다.** `agents/seed-architect.md`는 interview conversation에서 GOAL, CONSTRAINTS, ACCEPTANCE_CRITERIA, ONTOLOGY, EVALUATION_PRINCIPLES, EXIT_CONDITIONS, BROWNFIELD CONTEXT를 추출하고 exact output structure로 답하라고 지시한다([src/ouroboros/agents/seed-architect.md#L1-L71](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/agents/seed-architect.md#L1-L71)). `agents/evaluator.md`는 Stage 1 mechanical, Stage 2 semantic, Stage 3 consensus를 prompt-level procedure로 설명하고, final decision output format을 Markdown block으로 강제한다([src/ouroboros/agents/evaluator.md#L1-L75](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/agents/evaluator.md#L1-L75)).

**Auto/Run/Ralph skills는 MCP 호출을 직접 모방하지 말라는 프롬프트 경계를 둔다.** `skills/auto/SKILL.md`는 auto pipeline을 수동으로 흉내내지 말고 `ouroboros_start_auto` MCP tool을 호출하라고 하며, background job id와 auto_session_id를 보관하고 job wait/status/result로 terminal state까지 추적하라고 지시한다([skills/auto/SKILL.md#L20-L45](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/skills/auto/SKILL.md#L20-L45)). `skills/run/SKILL.md`는 seed execution MCP tools를 deferred tool discovery로 먼저 load하고, background execution starter, job relay loop, compact progress policy를 절차화한다([skills/run/SKILL.md#L34-L47](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/skills/run/SKILL.md#L34-L47), [skills/run/SKILL.md#L61-L111](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/skills/run/SKILL.md#L61-L111)). `skills/ralph/SKILL.md`는 natural-language request를 direct Ralph input으로 쓰지 말고 interview/seed path를 통과시키라고 지시한다([skills/ralph/SKILL.md#L68-L98](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/skills/ralph/SKILL.md#L68-L98)).

## 아키텍처

framework.md 정의: [아키텍처](../framework.md#4-아키텍처-architecture).

**배포 골격은 Python package `ouroboros-ai`와 Typer CLI다.** `pyproject.toml`은 project name을 `ouroboros-ai`, description을 “Specification-first workflow engine for AI coding agents. Works with Claude Code and Codex CLI.”, Python requirement를 `>=3.12`로 둔다([pyproject.toml#L1-L9](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/pyproject.toml#L1-L9)). core dependencies에는 aiosqlite, anyio, click, jsonschema, pydantic, prompt-toolkit, pyyaml, rich, sqlalchemy[asyncio], structlog, typer가 들어간다([pyproject.toml#L10-L22](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/pyproject.toml#L10-L22)). console script는 `ouroboros = ouroboros.cli.main:app`로 고정되어 있다([pyproject.toml#L38-L39](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/pyproject.toml#L38-L39)).

**Wheel packaging은 runtime assets를 package data로 포함한다.** `pyproject.toml`은 packages를 `src/ouroboros`로 두고, wheel force-include로 root `skills`, opencode plugin asset, plugin schemas, execution profile YAML을 각각 `ouroboros/skills`, `ouroboros/opencode/plugin`, `ouroboros/plugin/schemas`, `ouroboros/profiles`에 포함한다([pyproject.toml#L69-L95](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/pyproject.toml#L69-L95)). optional dependencies도 Claude, LiteLLM, MCP, TUI를 별도 extra로 나누며, exact pinning 주석은 fresh install이 compromised latest를 자동으로 끌지 않게 하려는 목적을 적는다([pyproject.toml#L24-L36](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/pyproject.toml#L24-L36)).

**소스 모듈 구조는 workflow OS의 phase boundary를 드러낸다.** README의 architecture overview는 `bigbang`, `routing`, `execution`, `evaluation`, `evolution`, `resilience`, `observability`, `persistence`, `orchestrator`, `core`, `providers`, `mcp`, `plugin`, `tui`, `cli`를 나열한다([README.md#L323-L345](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/README.md#L323-L345)). project-context도 Big Bang, PAL Router, Execution, Resilience, Evaluation, Consensus 여섯 phase를 core philosophy로 제시하고, CLI/Application/Domain/Infrastructure layered dependency 방향을 적는다([project-context.md#L5-L15](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/project-context.md#L5-L15), [project-context.md#L75-L89](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/project-context.md#L75-L89)).

**Event sourcing은 architectural state strategy다.** project-context는 all state changes via events라고 적고, BaseEvent shape와 event naming convention을 예시한다([project-context.md#L131-L153](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/project-context.md#L131-L153)). `workflow_lifecycle.py`도 Workflow IR lifecycle records를 event로 persist하고 later project할 수 있게 하는 narrow slice라고 설명하며, lifecycle event vocabulary를 `workflow.run.created`, `workflow.node.started`, `workflow.node.completed`, `workflow.checkpoint.saved`, `workflow.run.completed` 등으로 제한한다([src/ouroboros/orchestrator/workflow_lifecycle.py#L1-L47](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/orchestrator/workflow_lifecycle.py#L1-L47), [src/ouroboros/orchestrator/workflow_lifecycle.py#L89-L128](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/orchestrator/workflow_lifecycle.py#L89-L128)). lifecycle event model은 reason_code, refs, data, timestamp를 validation하고 `to_base_event`로 EventStore aggregate family에 묶는다([src/ouroboros/orchestrator/workflow_lifecycle.py#L519-L652](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/orchestrator/workflow_lifecycle.py#L519-L652)).

**Architecture의 특이점은 spec-first OS와 plugin firewall이 같은 repo에 공존한다는 점이다.** README는 OS layer로 Seed·Ledger·Runtime·MCP·safety boundaries를 제시하고([README.md#L55-L60](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/README.md#L55-L60)), CLI는 unknown top-level command를 plugin dispatch로 넘긴다([src/ouroboros/cli/main.py#L49-L72](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/cli/main.py#L49-L72)). plugin manifest loader는 remote loading이나 side effects를 하지 않고 validated manifest read-model에 머물러 있으며, trust store는 artifact digest와 exact scope grant를 별도로 보관한다([src/ouroboros/plugin/manifest.py#L20-L23](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/plugin/manifest.py#L20-L23), [src/ouroboros/plugin/trust_store.py#L83-L136](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/plugin/trust_store.py#L83-L136)).

## 모델별 튜닝

framework.md 정의: [모델별 튜닝](../framework.md#5-모델별-튜닝-per-model-tuning).

**PAL Router는 task complexity를 model tier로 매핑한다.** `routing/router.py`는 PAL Router가 task complexity로 어느 model tier가 task를 처리할지 결정한다고 설명하고, complexity < 0.4는 Frugal, 0.4–0.7은 Standard, 0.7 초과는 Frontier로 라우팅한다고 적는다([src/ouroboros/routing/router.py#L1-L10](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/router.py#L1-L10), [src/ouroboros/routing/router.py#L51-L91](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/router.py#L51-L91)). `complexity.py`는 token count 30%, tool dependency count 30%, AC nesting depth 40%를 complexity factors로 둔다([src/ouroboros/routing/complexity.py#L1-L11](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/complexity.py#L1-L11), [src/ouroboros/routing/complexity.py#L39-L52](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/complexity.py#L39-L52)).

**Tier model은 cost multiplier와 configured model list를 함께 검증한다.** `routing/tiers.py`는 Frugal 1x, Standard 10x, Frontier 30x cost system을 설명하고([src/ouroboros/routing/tiers.py#L1-L10](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/tiers.py#L1-L10)), `Tier.cost_multiplier`는 1/10/30을 반환한다([src/ouroboros/routing/tiers.py#L38-L64](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/tiers.py#L38-L64)). `get_tier_config`는 tier 존재 여부, configured models, expected cost factor를 검증하고, `get_model_for_tier`는 tier의 configured model 중 하나를 선택한다([src/ouroboros/routing/tiers.py#L66-L153](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/tiers.py#L66-L153), [src/ouroboros/routing/tiers.py#L156-L198](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/tiers.py#L156-L198)).

**Failure와 success 모두 tier movement를 만든다.** `routing/escalation.py`는 consecutive failure 2회가 Frugal → Standard → Frontier escalation을 만들고, Frontier 이후 실패는 resilience system을 위한 stagnation event로 이어진다고 적는다([src/ouroboros/routing/escalation.py#L1-L13](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/escalation.py#L1-L13), [src/ouroboros/routing/escalation.py#L44-L46](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/escalation.py#L44-L46)). `routing/downgrade.py`는 consecutive success 5회가 Frontier → Standard → Frugal downgrade를 만들고, 80% similarity 이상인 similar patterns가 tier preference를 상속한다고 적는다([src/ouroboros/routing/downgrade.py#L1-L16](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/downgrade.py#L1-L16), [src/ouroboros/routing/downgrade.py#L48-L50](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/downgrade.py#L48-L50)).

**Provider factory는 runtime별 adapter와 permission envelope 차이를 모델·런타임 분기로 처리한다.** `providers/factory.py`는 Claude Code, Codex CLI, Copilot CLI, Gemini CLI, Goose, OpenCode, Pi adapter를 import하고 backend name을 resolve한다([src/ouroboros/providers/factory.py#L11-L30](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/providers/factory.py#L11-L30), [src/ouroboros/providers/factory.py#L70-L80](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/providers/factory.py#L70-L80)). 같은 파일은 Gemini와 OpenCode처럼 hard allowed-tools surface가 없는 backend는 allowed_tools envelope를 prompt directive와 post-hoc violation event로 soft-enforce한다고 주석으로 구분하고, Claude Code와 Codex는 hard enforcement path에 남긴다([src/ouroboros/providers/factory.py#L38-L67](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/providers/factory.py#L38-L67)). `create_llm_adapter`는 backend별 adapter constructor와 runtime profile, permission mode, allowed_tools, max_turns, retries를 분기한다([src/ouroboros/providers/factory.py#L114-L243](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/providers/factory.py#L114-L243), [src/ouroboros/providers/factory.py#L243-L292](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/providers/factory.py#L243-L292)).

**Runtime profile YAML은 작업 성격별 model tier hint와 evidence schema를 분리한다.** `profiles/code.yaml`은 `axis: testable_unit`, runnable test, tests_passed/files_touched, subprocess_test_runner, suggested_model_tier medium을 둔다([src/ouroboros/profiles/code.yaml#L1-L28](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/profiles/code.yaml#L1-L28)). `profiles/analysis.yaml`은 perspective 비교와 tradeoff evidence를 요구하고, `profiles/research.yaml`은 independently cited sources와 triangulated_sources를 요구한다([src/ouroboros/profiles/analysis.yaml#L1-L27](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/profiles/analysis.yaml#L1-L27), [src/ouroboros/profiles/research.yaml#L1-L29](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/profiles/research.yaml#L1-L29)). 이는 code/analysis/research 작업이 같은 loop 안에서도 서로 다른 verification focus와 model tier hint를 갖게 하는 튜닝 표면이다.

**Consensus는 model diversity와 fallback perspective를 모두 가진다.** `evaluation/consensus.py`는 simple consensus가 3 models independent evaluation과 2/3 majority를 쓰고, deliberative consensus가 Advocate, Devil's Advocate, Judge role을 둔다고 설명한다([src/ouroboros/evaluation/consensus.py#L1-L18](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/evaluation/consensus.py#L1-L18)). default consensus models는 config에서 가져오며, OpenRouter API key가 없으면 same underlying model을 advocate/devil-advocate/judge perspective로 돌리는 single-model multi-perspective mode로 전환한다고 코드가 적는다([src/ouroboros/evaluation/consensus.py#L52-L90](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/evaluation/consensus.py#L52-L90), [src/ouroboros/evaluation/consensus.py#L246-L255](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/evaluation/consensus.py#L246-L255)).

## 설계 결정 요약

1. **Spec-first OS 정체성:** README가 Ouroboros를 local-first Agent OS와 replayable execution contract로 정의하고, ad-hoc prompting을 interview/crystallize/execute/evaluate/evolve로 대체한다고 설명한다([README.md#L39-L45](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/README.md#L39-L45)).
