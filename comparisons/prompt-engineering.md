# 프롬프트 엔지니어링 횡단 비교

## 차원 정의

[framework.md의 프롬프트 엔지니어링 정의](../framework.md#3-프롬프트-엔지니어링-prompt-engineering)는 모델에게 주입되는 자연어 지시문의 내용과 구조를 가리킨다. 시스템 프롬프트, 도구 설명문, 스킬·커맨드 Markdown, 역할 에이전트 페르소나, 키워드 트리거가 주입하는 지시문, 출력 형식 계약이 이 차원의 관찰 대상이다.

## 6개 하네스의 설계 분기

### opencode: `.txt` prompt asset과 synthetic reminder

opencode는 시스템 프롬프트와 도구 설명문을 TypeScript 코드가 아니라 번들된 `.txt` 자산으로 둔다. 모델 계열별 `session/prompt/*.txt`와 `tool/*.txt`가 분리되어 있고, 최종 system 배열은 에이전트 커스텀 프롬프트, 환경·지침, 사용자 system을 순서대로 결합한다([opencode 프롬프트](../harnesses/opencode.md#프롬프트-엔지니어링), [read.txt#L10](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/tool/read.txt#L10), [request.ts#L58-L66](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/session/llm/request.ts#L58-L66)). plan/build 전환은 system prompt 재작성 대신 synthetic user part로 표현되므로, 모드 상태를 대화 안의 reminder로 주입하는 형태다([opencode 프롬프트](../harnesses/opencode.md#프롬프트-엔지니어링), [reminders.ts#L33-L44](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/session/reminders.ts#L33-L44)).

### omo: 런타임 합성 페르소나와 호출자 대상 위임 프롬프트

omo는 정적 prompt asset보다 런타임 합성 프롬프트의 비중이 크다. 키워드 훅은 사용자 메시지에서 ultrawork 계열 trigger를 감지해 지시문을 주입하고, `default_mode.ultrawork`가 켜졌을 때는 system prompt에 `<ultrawork-mode>` 블록을 넣는다([omo 프롬프트](../harnesses/omo.md#프롬프트-엔지니어링), [keyword-detector/hook.ts#L96-L115](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/keyword-detector/hook.ts#L96-L115), [system-transform.ts#L26-L36](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/plugin/system-transform.ts#L26-L36)). 빌트인 에이전트 로스터는 팩토리 함수가 현재 도구·스킬·카테고리·동료 에이전트 메타데이터를 받아 delegation 표와 trigger 절을 생성하므로, 프롬프트가 코드-합성 산출물이 된다([omo 프롬프트](../harnesses/omo.md#프롬프트-엔지니어링), [builtin-agents.ts#L32-L59](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/agents/builtin-agents.ts#L32-L59), [sisyphus-agent-factory.ts#L32-L43](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/agents/sisyphus-agent-factory.ts#L32-L43)). 위임 카테고리의 `prompt_append`는 수신자가 아니라 호출자에게 “어떤 정보를 넣어야 하는가”를 지시한다는 점에서, 위임 품질을 발신 프롬프트 구조로 제어한다([omo 프롬프트](../harnesses/omo.md#프롬프트-엔지니어링), [anthropic-categories.ts#L6-L24](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/delegate-task/anthropic-categories.ts#L6-L24)).

### lazy-codex: OpenCode 스킬을 Codex 해석 규칙으로 기계 이식

lazy-codex의 prompt 설계는 공유 스킬을 손으로 다시 쓰기보다, build-time 변환으로 Codex 호환 해석층을 앞에 붙이는 방식이다. `sync-skills.mjs`는 OpenCode 전용 orchestration call을 감지하면 “Codex Harness Tool Compatibility” 절을 주입하고, 아래 코드 블록이 그 절과 충돌하면 compatibility 절이 우선한다고 선언한다([lazy-codex 프롬프트](../harnesses/lazycodex.md#프롬프트-엔지니어링), [sync-skills.mjs#L19-L39](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/sync-skills.mjs#L19-L39), [sync-skills.mjs#L80-L100](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/sync-skills.mjs#L80-L100)). ulw-loop·ulw-plan 스킬은 작은 SKILL.md와 `references/full-workflow.md`를 나눠 필요한 절만 읽게 하고, start-work/review-work는 문자열 치환으로 Codex판 gate 절을 깊게 덧붙인다([lazy-codex 프롬프트](../harnesses/lazycodex.md#프롬프트-엔지니어링), [ulw-loop/SKILL.md#L12](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/ulw-loop/SKILL.md#L12), [sync-skills.mjs#L109-L141](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/sync-skills.mjs#L109-L141)).

### omc: 스킬 Markdown과 역할 파일을 Claude Code 세션 지시문으로 렌더링

omc는 `skills/` 디렉터리를 번들 SSOT로 읽어 frontmatter, alias, pipeline metadata, runtime guidance를 skill entry로 만들고, Claude Code native command 이름과 충돌하는 short name에는 `omc-` prefix를 붙인다([omc 프롬프트](../harnesses/omc.md#프롬프트-엔지니어링), [src/features/builtin-skills/skills.ts#L1-L20](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/builtin-skills/skills.ts#L1-L20), [src/features/builtin-skills/skills.ts#L54-L85](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/builtin-skills/skills.ts#L54-L85)). keyword-detector는 ultrawork, ralplan, deep interview, code review 같은 자연어 trigger를 모드 지시문으로 바꾸고, code block·XML tag·URL·file path를 걷어내는 sanitizer로 오탐을 줄인다([omc 프롬프트](../harnesses/omc.md#프롬프트-엔지니어링), [keyword-detector.mjs#L3-L24](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/keyword-detector.mjs#L3-L24), [keyword-detector.mjs#L212-L263](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/keyword-detector.mjs#L212-L263)). 역할 에이전트 파일은 model frontmatter, read-only/implementation boundary, 성공 조건을 자연어로 고정한다([omc 프롬프트](../harnesses/omc.md#프롬프트-엔지니어링), [agents/architect.md#L1-L36](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/architect.md#L1-L36), [agents/executor.md#L20-L40](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/executor.md#L20-L40)).

### gajae-code: workflow skill과 role-agent prompt를 source-bundled contract로 고정

GJC는 네 workflow skill과 네 role-agent prompt를 source-bundled Markdown으로 고정하고, runtime `.gjc` copy가 아니라 source path를 SSOT로 둔다([gajae-code 프롬프트](../harnesses/gajae-code.md#프롬프트-엔지니어링), [README.md#L141-L146](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/README.md#L141-L146), [AGENTS.md#L5-L15](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/AGENTS.md#L5-L15)). Deep Interview prompt는 Socratic questioning, mathematical ambiguity scoring, one-question rule, topology gate, internal auto-mode fragment boundary를 긴 절차문으로 명시하고, Ralplan prompt는 Planner/Architect/Critic loop와 receipt-only artifact discipline을 모델에게 요구한다([gajae-code 프롬프트](../harnesses/gajae-code.md#프롬프트-엔지니어링), [deep-interview/SKILL.md#L13-L15](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L13-L15), [deep-interview/SKILL.md#L58-L64](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L58-L64), [ralplan/SKILL.md#L40-L68](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/ralplan/SKILL.md#L40-L68)). System prompt 조립도 project/user SYSTEM.md 우선순위와 tool metadata map으로 코드화되어 있어, source prompt와 runtime context가 함께 들어간다([gajae-code 프롬프트](../harnesses/gajae-code.md#프롬프트-엔지니어링), [system-prompt.ts#L277-L326](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/system-prompt.ts#L277-L326)).

### ouroboros: “질문만 하는” interviewer와 Seed/Evaluator prompt

Ouroboros의 prompt 설계는 스펙 주도 workflow의 각 역할을 자연어 contract로 나눈다. `skills/interview/SKILL.md`는 interview path, MCP discovery, lateral review dispatch를 절차화하고, Socratic interviewer prompt는 “ONLY an interviewer”, “NEVER promise to build demos, write code, or execute anything”라는 역할 경계를 반복한다([ouroboros 프롬프트](../harnesses/ouroboros.md#프롬프트-엔지니어링), [skills/interview/SKILL.md#L15-L36](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/skills/interview/SKILL.md#L15-L36), [socratic-interviewer.md#L1-L10](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/agents/socratic-interviewer.md#L1-L10)). Persona prompts는 researcher, simplifier, architect, breadth-keeper, seed-closer 같은 perspective switch를 제공하고, Seed architect와 Evaluator prompt는 output structure와 evaluation stages를 Markdown contract로 강제한다([ouroboros 프롬프트](../harnesses/ouroboros.md#프롬프트-엔지니어링), [interview.py#L90-L136](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/interview.py#L90-L136), [seed-architect.md#L1-L71](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/agents/seed-architect.md#L1-L71), [evaluator.md#L1-L75](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/agents/evaluator.md#L1-L75)).

## 분기 원인 분석

첫 번째 원인은 **프롬프트가 코드와 얼마나 가까운가**다. opencode는 `.txt` asset을 코드 옆에 두고, omo는 factory 함수가 runtime metadata를 받아 prompt를 합성한다([opencode 프롬프트](../harnesses/opencode.md#프롬프트-엔지니어링), [omo 프롬프트](../harnesses/omo.md#프롬프트-엔지니어링)). lazy-codex는 build script가 compatibility overlay를 삽입하고, omc·GJC·Ouroboros는 Markdown skill을 workflow contract로 삼는다([lazy-codex 프롬프트](../harnesses/lazycodex.md#프롬프트-엔지니어링), [omc 프롬프트](../harnesses/omc.md#프롬프트-엔지니어링), [gajae-code 프롬프트](../harnesses/gajae-code.md#프롬프트-엔지니어링), [ouroboros 프롬프트](../harnesses/ouroboros.md#프롬프트-엔지니어링)).

두 번째 원인은 **역할 경계의 표현 방식**이다. opencode는 agent prompt와 permission rule이 함께 역할을 만든다([opencode 아키텍처](../harnesses/opencode.md#아키텍처)). omo·omc·GJC는 역할 에이전트 프롬프트가 persona, 도구 사용, 검증 책임을 담고, Ouroboros는 interviewer/seed/evaluator persona를 분리해 “질문”, “스펙 생성”, “평가”의 언어 contract를 나눈다([omo 프롬프트](../harnesses/omo.md#프롬프트-엔지니어링), [omc 프롬프트](../harnesses/omc.md#프롬프트-엔지니어링), [gajae-code 프롬프트](../harnesses/gajae-code.md#프롬프트-엔지니어링), [ouroboros 프롬프트](../harnesses/ouroboros.md#프롬프트-엔지니어링)).

세 번째 원인은 **host 호환성의 처리 위치**다. lazy-codex는 OpenCode용 skill 본문을 유지한 채 Codex compatibility 절을 앞에 붙이는 오버레이 방식을 택하고, GJC는 source default/runtime state 분리를 prompt surface에 명시해 GJC entrypoint만 public workflow로 둔다([lazy-codex 프롬프트](../harnesses/lazycodex.md#프롬프트-엔지니어링), [gajae-code 프롬프트](../harnesses/gajae-code.md#프롬프트-엔지니어링)). omc는 Claude Code command/skill 충돌을 loader에서 prefix로 처리하고, Ouroboros는 여러 runtime을 target하면서도 Seed와 MCP tool boundary를 skill text에 반복한다([omc 프롬프트](../harnesses/omc.md#프롬프트-엔지니어링), [ouroboros 프롬프트](../harnesses/ouroboros.md#프롬프트-엔지니어링)).

## 비교 메모

프롬프트 엔지니어링의 공통 질문은 “모델이 무엇을 자기 임무로 이해하게 만드는가”다. opencode는 모델별 `.txt`와 reminder, omo는 runtime 합성 persona, lazy-codex는 compatibility overlay, omc는 Claude Code skill/agent Markdown, GJC는 source-bundled workflow contract, Ouroboros는 spec-first persona chain으로 답한다. 하네스가 복잡해질수록 프롬프트는 단일 system message보다 workflow state와 role boundary를 설명하는 실행 문서에 가까워진다.
