# 모델별 튜닝 횡단 비교

## 차원 정의

[framework.md의 모델별 튜닝 정의](../framework.md#5-모델별-튜닝-per-model-tuning)는 특정 모델·모델 계열의 성질에 맞춘 분기와 최적화를 가리킨다. 모델 라우팅, 폴백 체인, 모델별 프롬프트 변형, 토큰 예산·컨텍스트 윈도 전략, 모델별 도구 호출 형식 대응, 비용·쿼터 규율이 이 차원의 비교 대상이다.

## 6개 하네스의 설계 분기

### opencode: 모델 ID 부분 문자열이 prompt·tool·provider transform을 바꾼다

opencode는 모델 ID 부분 문자열로 시스템 프롬프트 파일을 고른다. gpt-4/o1/o3, codex, 기타 gpt, gemini, claude, trinity, kimi, default 계열이 각각 다른 prompt 자산으로 이어진다([opencode 모델별 튜닝](../harnesses/opencode.md#모델별-튜닝), [system.ts#L25-L39](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/session/system.ts#L25-L39)). 도구 표면도 모델에 따라 바뀐다. GPT-5 계열에는 edit/write 대신 apply_patch를 노출하고, 그 외 모델에는 edit/write를 노출한다([opencode 모델별 튜닝](../harnesses/opencode.md#모델별-튜닝), [registry.ts#L273-L276](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/tool/registry.ts#L273-L276)). ProviderTransform은 reasoning effort, thinking budget, temperature/topP/topK, prompt caching marker, output token cap, provider별 tool-call workaround를 요청 직전에 흡수한다([opencode 모델별 튜닝](../harnesses/opencode.md#모델별-튜닝), [transform.ts#L479-L541](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/provider/transform.ts#L479-L541), [llm.ts#L325-L337](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/session/llm.ts#L325-L337)).

### omo: 작업 카테고리와 에이전트 페르소나가 모델 선택을 데이터화한다

omo는 모델별 튜닝을 메인 에이전트 prompt, 작업 카테고리, 에이전트별 폴백 체인에 동시에 배치한다. sisyphus prompt는 Claude, GPT, Gemini, Kimi 등 계열별 파일로 나뉘고, factory가 model ID 판별 함수로 해당 변형을 고른다([omo 모델별 튜닝](../harnesses/omo.md#모델별-튜닝), [sisyphus/index.ts#L1-L12](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/agents/sisyphus/index.ts#L1-L12), [sisyphus-agent-factory.ts#L45-L91](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/agents/sisyphus-agent-factory.ts#L45-L91)). 작업 카테고리는 visual-engineering, ultrabrain, deep, artistry, quick, unspecified-low/high, writing으로 나뉘고, 각 항목은 model, variant, thinking, reasoningEffort, temperature를 함께 둔다([omo 모델별 튜닝](../harnesses/omo.md#모델별-튜닝), [categories.ts#L4-L39](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/config/schema/categories.ts#L4-L39), [builtin-categories.ts#L8-L13](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/delegate-task/builtin-categories.ts#L8-L13)). model-core에는 에이전트별 provider/model/variant 우선순위 리스트가 있어, 역할 단위 폴백 체인도 코드화된다([omo 모델별 튜닝](../harnesses/omo.md#모델별-튜닝), [agent-model-requirements.ts#L3-L31](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/model-core/src/agent-model-requirements.ts#L3-L31)).

### lazy-codex: 단일 provider 안에서 역할별 OpenAI 모델과 effort를 나눈다

lazy-codex는 omo의 멀티 provider 라우팅을 Codex 환경에 맞게 OpenAI 계열 내부의 역할별 등급 분기로 좁힌다. 역할 TOML은 explorer/librarian에 `gpt-5.4-mini`와 low effort/fast tier를 주고, plan/momus/metis/reviewer에는 `gpt-5.5`와 high 또는 xhigh effort를 준다([lazy-codex 모델별 튜닝](../harnesses/lazycodex.md#모델별-튜닝), [explorer.toml#L4-L6](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/agents/explorer.toml#L4-L6), [plan.toml#L4-L5](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/agents/plan.toml#L4-L5), [codex-ultrawork-reviewer.toml#L4-L5](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/agents/codex-ultrawork-reviewer.toml#L4-L5)). `model-catalog.json`은 현재 프로필과 managed legacy profiles를 배포하고, config migration은 사용자가 바꾸지 않은 관리 구역만 새 프로필로 옮긴다([lazy-codex 모델별 튜닝](../harnesses/lazycodex.md#모델별-튜닝), [model-catalog.json#L2-L42](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/model-catalog.json#L2-L42), [migrate-codex-config.mjs#L69-L80](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/scripts/migrate-codex-config.mjs#L69-L80)).

### omc: 요청 신호·위임 카테고리·역할 파일이 tier 선택을 중복 기록한다

omc는 LOW/MEDIUM/HIGH tier를 중심에 둔 모델 라우터를 가진다. `routeTask`는 explicit model과 agent override를 먼저 본 뒤 prompt signal, rules, scorer를 결합하고, rules와 scorer가 크게 벌어지면 높은 tier를 택한다([omc 모델별 튜닝](../harnesses/omc.md#모델별-튜닝), [router.ts#L22-L64](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/model-routing/router.ts#L22-L64), [router.ts#L71-L113](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/model-routing/router.ts#L71-L113)). Signal extractor는 word count, file path count, code block count, architecture/debug/risk keyword, estimated subtasks, cross-file dependency, reversibility, impact scope를 분리한다([omc 모델별 튜닝](../harnesses/omc.md#모델별-튜닝), [signals.ts#L17-L53](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/model-routing/signals.ts#L17-L53), [signals.ts#L85-L213](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/model-routing/signals.ts#L85-L213)). 같은 tier 선택은 delegation category의 temperature/thinkingBudget와 역할 파일의 haiku/sonnet/opus frontmatter에도 반복된다([omc 모델별 튜닝](../harnesses/omc.md#모델별-튜닝), [delegation-categories/index.ts#L31-L82](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/features/delegation-categories/index.ts#L31-L82), [agents/explore.md#L1-L7](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/explore.md#L1-L7), [agents/architect.md#L1-L7](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/agents/architect.md#L1-L7)).

### gajae-code: provider/model selector, thinking suffix, role profile binding

GJC는 provider/model selector와 thinking-level suffix를 parse해 `provider/model:level` 형태를 내부 모델 선택 단위로 쓴다([gajae-code 모델별 튜닝](../harnesses/gajae-code.md#모델별-튜닝), [model-resolver.ts#L35-L67](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/config/model-resolver.ts#L35-L67)). OpenRouter만 routed/date suffix를 벗긴 fallback candidate queue와 bare model id ambiguity resolution을 갖고, 다른 provider는 exact match 중심으로 해석된다([gajae-code 모델별 튜닝](../harnesses/gajae-code.md#모델별-튜닝), [model-resolver.ts#L69-L180](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/config/model-resolver.ts#L69-L180), [model-resolver.ts#L197-L267](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/config/model-resolver.ts#L197-L267)). Built-in profile은 default, executor, architect, planner, critic selector를 따로 매핑하고, role prompt frontmatter도 thinking level을 가진다([gajae-code 모델별 튜닝](../harnesses/gajae-code.md#모델별-튜닝), [model-profiles.ts#L56-L120](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/config/model-profiles.ts#L56-L120), [prompts/agents/architect.md#L1-L10](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/prompts/agents/architect.md#L1-L10)).

### ouroboros: complexity tier, escalation/downgrade, provider adapter envelope

Ouroboros는 PAL Router가 task complexity를 Frugal, Standard, Frontier tier로 매핑한다. Complexity는 token count, tool dependency count, AC nesting depth를 가중 요소로 두며, router는 0.4/0.7 경계를 기준으로 tier를 고른다([ouroboros 모델별 튜닝](../harnesses/ouroboros.md#모델별-튜닝), [complexity.py#L39-L52](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/complexity.py#L39-L52), [router.py#L51-L91](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/router.py#L51-L91)). Tier model은 cost multiplier와 configured model list를 함께 검증하고, failure 2회는 escalation, success 5회는 downgrade로 이어진다([ouroboros 모델별 튜닝](../harnesses/ouroboros.md#모델별-튜닝), [tiers.py#L38-L64](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/tiers.py#L38-L64), [escalation.py#L1-L13](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/escalation.py#L1-L13), [downgrade.py#L1-L16](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/routing/downgrade.py#L1-L16)). Provider factory는 Claude Code, Codex CLI, Copilot CLI, Gemini CLI, Goose, OpenCode, Pi adapter를 backend별로 만들고, allowed-tools hard/soft enforcement 차이를 runtime profile과 permission envelope로 처리한다([ouroboros 모델별 튜닝](../harnesses/ouroboros.md#모델별-튜닝), [factory.py#L38-L80](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/providers/factory.py#L38-L80), [factory.py#L114-L292](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/providers/factory.py#L114-L292)).

## 분기 원인 분석

첫 번째 원인은 **모델 선택이 사용자-facing 설정인가, 내부 자동화인가**다. opencode는 모델 ID 문자열에서 prompt와 도구 표면을 즉시 분기하고, GJC는 provider/model selector와 role profile로 사용자가 읽을 수 있는 binding을 둔다([opencode 모델별 튜닝](../harnesses/opencode.md#모델별-튜닝), [gajae-code 모델별 튜닝](../harnesses/gajae-code.md#모델별-튜닝)). omo·omc·Ouroboros는 작업 분류와 역할·complexity tier가 모델 선택을 움직이는 자동화 층을 더 많이 둔다([omo 모델별 튜닝](../harnesses/omo.md#모델별-튜닝), [omc 모델별 튜닝](../harnesses/omc.md#모델별-튜닝), [ouroboros 모델별 튜닝](../harnesses/ouroboros.md#모델별-튜닝)).

두 번째 원인은 **provider 다양성**이다. lazy-codex는 Codex 환경의 OpenAI 계열 안에서 역할별 effort를 나누고, opencode·omo·GJC·Ouroboros는 provider별 transform, fallback, adapter, profile mapping을 갖는다([lazy-codex 모델별 튜닝](../harnesses/lazycodex.md#모델별-튜닝), [opencode 모델별 튜닝](../harnesses/opencode.md#모델별-튜닝), [omo 모델별 튜닝](../harnesses/omo.md#모델별-튜닝), [gajae-code 모델별 튜닝](../harnesses/gajae-code.md#모델별-튜닝), [ouroboros 모델별 튜닝](../harnesses/ouroboros.md#모델별-튜닝)). omc는 Claude Code 안의 haiku/sonnet/opus tier와 외부 CLI worker별 실행 인자를 함께 다룬다([omc 모델별 튜닝](../harnesses/omc.md#모델별-튜닝)).

세 번째 원인은 **비용·용량·실패를 어느 시점에 반영하는가**다. opencode는 ProviderTransform에서 요청 파라미터와 output token cap을 조정하고, lazy-codex는 model catalog와 migration으로 배포 시점 설정을 조정한다([opencode 모델별 튜닝](../harnesses/opencode.md#모델별-튜닝), [lazy-codex 모델별 튜닝](../harnesses/lazycodex.md#모델별-튜닝)). omc와 Ouroboros는 prompt signal·task complexity·failure/success history를 runtime routing에 반영하고, GJC는 retry budget과 role profile binding으로 provider stream 특성을 운영 규칙에 넣는다([omc 모델별 튜닝](../harnesses/omc.md#모델별-튜닝), [ouroboros 모델별 튜닝](../harnesses/ouroboros.md#모델별-튜닝), [gajae-code 모델별 튜닝](../harnesses/gajae-code.md#모델별-튜닝)).

## 비교 메모

모델별 튜닝의 공통 질문은 “어떤 일에 어떤 모델 계열과 실행 파라미터를 붙일 것인가”다. opencode는 모델 문자열 기반 prompt/tool/provider transform, omo는 카테고리와 에이전트 폴백 체인, lazy-codex는 OpenAI 역할 TOML과 catalog, omc는 prompt signal과 tier router, GJC는 selector/profile/thinking suffix, Ouroboros는 complexity tier와 adapter envelope로 답한다. 이 분기는 모델이 단순 backend가 아니라 하네스의 역할·비용·검증 구조를 결정하는 축임을 보여준다.
