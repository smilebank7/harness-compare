# 아키텍처 횡단 비교

## 차원 정의

[framework.md의 아키텍처 정의](../framework.md#4-아키텍처-architecture)는 하네스를 구성하는 방식 자체를 가리킨다. 배포 형태, 모듈 경계와 의존 방향, 상태 저장 위치, 확장점 설계, 베이스 플랫폼과의 결합도, 멀티 패키지·멀티 배포판 전략이 이 차원의 비교 대상이다.

## 6개 하네스의 설계 분기

### opencode: 코어·플러그인 타입·SDK 경계가 만든 확장 골격

opencode의 아키텍처는 코어가 확장점을 소유하고, 플러그인이 공개 SDK 표면을 통해서만 내부 기능에 닿는 구조다. `Plugin = (input) => Promise<Hooks>` 계약은 SDK client, project info, Bun shell을 `PluginInput`으로 넘기며, SDK client는 네트워크가 아니라 in-process server fetch에 바인딩된다([opencode 아키텍처](../harnesses/opencode.md#아키텍처), [plugin/src/index.ts#L56-L74](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/plugin/src/index.ts#L56-L74), [plugin/index.ts#L139-L144](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/plugin/index.ts#L139-L144)). Hooks 객체는 tool 등록, chat params/header 개입, permission ask, tool execute before/after, system/compaction prompt transform, tool definition rewrite를 같은 `(input, output)` mutation pattern으로 묶는다([opencode 아키텍처](../harnesses/opencode.md#아키텍처), [plugin/src/index.ts#L222-L280](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/plugin/src/index.ts#L222-L280), [plugin/index.ts#L277-L291](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/plugin/index.ts#L277-L291)).

### omo: 플러그인 1장 뒤의 하네스 중립 코어 모노레포

omo는 opencode 플러그인 하나로 출하되지만, repo 내부는 `hashline-core`, `model-core`, `prompts-core`, `rules-engine`, `boulder-state`, `agents-md-core`와 `omo-opencode`/`omo-codex` 어댑터로 나뉜 workspace다([omo 아키텍처](../harnesses/omo.md#아키텍처), [package.json#L8-L23](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/package.json#L8-L23)). 하네스 중립 코어의 경계는 관례가 아니라 테스트로 강제된다. prompts-core에 `@opencode-ai` import가 있으면 coupling audit test가 실패한다([omo 아키텍처](../harnesses/omo.md#아키텍처), [opencode-coupling-audit.test.ts#L9-L13](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/prompts-core/test/opencode-coupling-audit.test.ts#L9-L13)). 이 구조 덕분에 Ultimate(opencode)와 Light(Codex) 에디션은 같은 코어를 공유하되, host별 표면만 다른 어댑터로 갈라진다([omo 아키텍처](../harnesses/omo.md#아키텍처), [plugin.json#L21-L23](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-codex/plugin/.codex-plugin/plugin.json#L21-L23)).

### lazy-codex: 배포 정체성이 제품의 중심인 thin distribution

lazy-codex는 제품 코드의 원천을 omo 쪽에 두고, 이 repo는 npm alias bin, Codex marketplace bundle, 웹사이트, 서브모듈 pin을 담는 배포 정체성으로 움직인다([lazy-codex 아키텍처](../harnesses/lazycodex.md#아키텍처), [README.md#L223-L232](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/README.md#L223-L232), [.gitmodules#L1-L3](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/.gitmodules#L1-L3)). `lazycodex-ai install`은 `npx --yes --package oh-my-openagent omo install --platform=codex`로 셸아웃하는 이름과 기본 플래그의 고정 표기이고, root test는 그 문자열과 package metadata를 단언한다([lazy-codex 아키텍처](../harnesses/lazycodex.md#아키텍처), [bin/lazycodex-ai.js#L8-L24](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/bin/lazycodex-ai.js#L8-L24), [lazycodex-ai-bin.test.mjs#L47-L63](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/test/lazycodex-ai-bin.test.mjs#L47-L63)). PR source guidance workflow가 이 repo의 변경을 닫고 upstream package path로 보내라는 안내를 남기는 점도 source-of-truth 경계를 CI가 집행하는 사례다([lazy-codex 아키텍처](../harnesses/lazycodex.md#아키텍처), [pr-source-guidance.yml#L26-L35](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/.github/workflows/pr-source-guidance.yml#L26-L35)).

### omc: Claude Code 플러그인·npm CLI·bridge·tmux runtime의 결합 패키지

omc는 한 repo가 Claude Code plugin manifest, npm CLI, bridge CJS, hooks, scripts, skills, templates, docs, MCP config를 함께 출하한다([omc 아키텍처](../harnesses/omc.md#아키텍처), [package.json#L19-L40](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/package.json#L19-L40), [package.json#L41-L50](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/package.json#L41-L50)). 세션 내 `/team`과 터미널 `omc team`은 같은 이름을 공유하지만 상태·통신·프로세스 모델이 다른 병렬 표면으로 설명된다([omc 아키텍처](../harnesses/omc.md#아키텍처), [README.md#L105-L118](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L105-L118), [README.md#L156-L184](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L156-L184)). 상태 루트는 `.omc`와 workspace marker 규칙으로 묶이고, Team runtime도 `.omc/state/team/<teamName>` 아래 task/worker state를 둔다([omc 아키텍처](../harnesses/omc.md#아키텍처), [README.md#L315-L328](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L315-L328), [src/team/runtime.ts#L115-L119](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/runtime.ts#L115-L119)).

### gajae-code: 독립형 `gjc` runner와 runtime `.gjc` 상태 경계

GJC는 root workspace와 `@gajae-code/coding-agent` package가 `gjc` CLI를 구성하는 독립형 runner다([gajae-code 아키텍처](../harnesses/gajae-code.md#아키텍처), [package.json#L1-L31](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/package.json#L1-L31), [packages/coding-agent/package.json#L28-L32](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/package.json#L28-L32)). Runtime state는 product source와 분리되어 `.gjc` 아래에 놓이고, workflow/agent definition은 source Markdown에서 load되며 runtime spec/plan/ledger만 `.gjc`에 쌓인다([gajae-code 아키텍처](../harnesses/gajae-code.md#아키텍처), [AGENTS.md#L23-L30](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/AGENTS.md#L23-L30), [README.md#L141-L146](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/README.md#L141-L146)). `packages/coding-agent/package.json` exports가 legacy MCP/command/autoresearch/ralph/ultraqa/ultrawork surfaces를 `null`로 닫는 점은 public architecture를 작은 workflow surface로 정리하는 경계다([gajae-code 아키텍처](../harnesses/gajae-code.md#아키텍처), [packages/coding-agent/package.json#L88-L147](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/package.json#L88-L147)).

### ouroboros: Python workflow OS와 plugin firewall의 공존

Ouroboros는 `ouroboros-ai` Python package와 Typer CLI를 골격으로 삼고, `src/ouroboros` package data에 skills, opencode plugin asset, plugin schemas, profiles를 wheel에 포함한다([ouroboros 아키텍처](../harnesses/ouroboros.md#아키텍처), [pyproject.toml#L1-L39](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/pyproject.toml#L1-L39), [pyproject.toml#L69-L95](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/pyproject.toml#L69-L95)). Source module 구조는 `bigbang`, `routing`, `execution`, `evaluation`, `evolution`, `resilience`, `observability`, `persistence`, `orchestrator`, `core`, `providers`, `mcp`, `plugin`, `tui`, `cli`로 phase boundary를 드러낸다([ouroboros 아키텍처](../harnesses/ouroboros.md#아키텍처), [README.md#L323-L345](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/README.md#L323-L345), [project-context.md#L75-L89](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/project-context.md#L75-L89)). 동시에 plugin manifest loader와 trust store는 외부 capability를 validated manifest, artifact digest, exact scope grant에 묶어 OS kernel과 plugin 확장 경계를 분리한다([ouroboros 아키텍처](../harnesses/ouroboros.md#아키텍처), [plugin/manifest.py#L20-L23](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/plugin/manifest.py#L20-L23), [plugin/trust_store.py#L83-L136](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/plugin/trust_store.py#L83-L136)).

## 분기 원인 분석

첫 번째 원인은 **베이스 플랫폼에 붙는 방식**이다. opencode는 자신이 host이면서 plugin API를 제공하고, omo는 그 API 위에 얹히며, lazy-codex는 Codex bundle/distribution으로 같은 의도를 옮긴다([opencode 아키텍처](../harnesses/opencode.md#아키텍처), [omo 아키텍처](../harnesses/omo.md#아키텍처), [lazy-codex 아키텍처](../harnesses/lazycodex.md#아키텍처)). omc는 Claude Code plugin과 별도 tmux CLI runtime을 한 패키지 안에서 병렬 배치하고, GJC·Ouroboros는 외부 runner/OS로 host 바깥에 선다([omc 아키텍처](../harnesses/omc.md#아키텍처), [gajae-code 아키텍처](../harnesses/gajae-code.md#아키텍처), [ouroboros 아키텍처](../harnesses/ouroboros.md#아키텍처)).

두 번째 원인은 **state root가 무엇을 의미하는가**다. opencode와 omo는 host config/plugin directory와 workspace package boundary가 중심이고, lazy-codex는 marketplace bundle과 installer alias의 동기화가 중심이다([opencode 아키텍처](../harnesses/opencode.md#아키텍처), [omo 아키텍처](../harnesses/omo.md#아키텍처), [lazy-codex 아키텍처](../harnesses/lazycodex.md#아키텍처)). omc와 GJC는 각각 `.omc`와 `.gjc`를 runtime state root로 삼아 team/plan/ledger를 저장하고, Ouroboros는 EventStore와 workflow lifecycle events를 architectural state strategy로 둔다([omc 아키텍처](../harnesses/omc.md#아키텍처), [gajae-code 아키텍처](../harnesses/gajae-code.md#아키텍처), [ouroboros 아키텍처](../harnesses/ouroboros.md#아키텍처)).

세 번째 원인은 **확장성을 어디에서 통제하는가**다. opencode는 Hooks interface와 SDK boundary, omo는 중립 코어와 adapter split, lazy-codex는 CI로 닫힌 source boundary, omc는 skill/bridge/team runtime의 multi-surface package, GJC는 public workflow surface와 null exports, Ouroboros는 plugin manifest/trust store로 확장성을 제어한다([opencode 아키텍처](../harnesses/opencode.md#아키텍처), [omo 아키텍처](../harnesses/omo.md#아키텍처), [lazy-codex 아키텍처](../harnesses/lazycodex.md#아키텍처), [omc 아키텍처](../harnesses/omc.md#아키텍처), [gajae-code 아키텍처](../harnesses/gajae-code.md#아키텍처), [ouroboros 아키텍처](../harnesses/ouroboros.md#아키텍처)).

## 비교 메모

아키텍처 차원의 공통 질문은 “기능들이 어디에 살고, 어떤 경계를 넘을 수 있는가”다. opencode는 plugin host, omo는 host-neutral core + adapter, lazy-codex는 distribution identity, omc는 plugin/CLI/bridge bundle, GJC는 external runner + `.gjc` runtime, Ouroboros는 Python workflow OS + plugin firewall로 답한다. 이 골격 차이가 하네스 표면, 루프 상태, 프롬프트 로딩, 모델 라우팅을 모두 제약한다.
