# 루프 엔지니어링 횡단 비교

## 차원 정의

[framework.md의 루프 엔지니어링 정의](../framework.md#2-루프-엔지니어링-loop-engineering)는 에이전트 실행의 시간 축을 제어하는 동적 설비를 가리킨다. 여기에는 반복 제어, 종료 조건, 재시도·복구, 검증 사이클, 체크포인트·이어가기, 서브에이전트 수명 주기가 포함된다.

## 6개 하네스의 설계 분기

### opencode: 한 턴 루프를 `compact | stop | continue`로 환원

opencode의 기본 루프는 `runLoop`의 `while (true)`이며, 한 스텝의 결과를 `compact`, `stop`, `continue` 셋으로 환원한다([opencode 루프](../harnesses/opencode.md#루프-엔지니어링), [prompt.ts#L1134-L1141](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/session/prompt.ts#L1134-L1141), [processor.ts#L36](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/session/processor.ts#L36)). 종료는 finish reason과 미처리 tool-call part의 이중 검사로 판단하고, 스텝 상한은 강제 kill이 아니라 `max-steps.txt` assistant message 주입으로 마무리를 유도한다([opencode 루프](../harnesses/opencode.md#루프-엔지니어링), [prompt.ts#L1159-L1169](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/session/prompt.ts#L1159-L1169), [prompt.ts#L1231-L1232](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/session/prompt.ts#L1231-L1232)). 같은 도구·같은 입력 3회 반복은 `doom_loop` 권한 질의로 넘어가므로, 반복 감지는 루프에 있지만 처분은 권한 경계로 라우팅된다([opencode 루프](../harnesses/opencode.md#루프-엔지니어링), [processor.ts#L522-L545](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/session/processor.ts#L522-L545)).

### omo: 완료 주장을 의심하고 세션을 다시 굴리는 플러그인 루프

omo는 opencode의 종료 기본값을 그대로 믿지 않는다. 세션이 idle 상태인데 미완료 todo가 남으면 `CONTINUATION_PROMPT`를 넣어 다시 실행시키고, 정체 3회·연속 실패 5회 상한과 복구·취소·토큰 한도·백그라운드 실행 조건을 skip guard로 둔다([omo 루프](../harnesses/omo.md#루프-엔지니어링), [constants.ts#L7-L14](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/todo-continuation-enforcer/constants.ts#L7-L14), [idle-event.ts#L42-L84](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/todo-continuation-enforcer/idle-event.ts#L42-L84), [constants.ts#L20-L25](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/todo-continuation-enforcer/constants.ts#L20-L25)). ralph-loop은 `startLoop`/`resumeLoop`/`cancelLoop` 인터페이스로 completionPromise가 충족될 때까지 같은 세션 프롬프트를 재투입하고, `delegate_task`는 동기 대기와 백그라운드 실행·continuation을 나눈다([omo 루프](../harnesses/omo.md#루프-엔지니어링), [ralph-loop-hook.ts#L10-L26](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/ralph-loop/ralph-loop-hook.ts#L10-L26), [delegate-task/tools.ts#L45-L54](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/delegate-task/tools.ts#L45-L54)).

### lazy-codex: Stop 훅과 파일 상태로 재구성한 Codex 루프

lazy-codex는 opencode plugin 내부 상태 대신 Codex Stop/SubagentStop 훅과 `.omo` 파일 상태를 루프의 판단 근거로 쓴다. start-work-continuation은 `.omo/boulder.json`에서 현재 세션 작업이 active/paused인지 보고, 미체크 TODO나 final verification wave가 남아 있으면 `decision: "block"`으로 종료를 거부한다([lazy-codex 루프](../harnesses/lazycodex.md#루프-엔지니어링), [start-work-continuation/src/codex-hook.ts#L6-L16](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/start-work-continuation/src/codex-hook.ts#L6-L16), [boulder-reader.ts#L60-L85](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/start-work-continuation/src/boulder-reader.ts#L60-L85)). ulw-loop 상태는 `.omo/ulw-loop/<session-id>/` 아래 brief·goals·ledger로 내려오고, 컴팩션 뒤에는 ledger부터 다시 읽으라는 skill 지시가 붙는다([lazy-codex 루프](../harnesses/lazycodex.md#루프-엔지니어링), [paths.ts#L8-L36](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ulw-loop/src/paths.ts#L8-L36), [ulw-loop/SKILL.md#L22-L24](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/skills/ulw-loop/SKILL.md#L22-L24)).

### omc: Team 단계 루프와 Stop 훅 persistent-mode

omc의 루프 중심은 Team이다. README와 Team skill은 `team-plan → team-prd → team-exec → team-verify → team-fix` 흐름을 공개 계약으로 두고, verify/fix는 terminal blocked/failed 또는 검증 통과까지 반복된다([omc 루프](../harnesses/omc.md#루프-엔지니어링), [README.md#L140-L142](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/README.md#L140-L142), [skills/team/SKILL.md#L147-L155](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/skills/team/SKILL.md#L147-L155)). Team runtime은 task status, owner, assigned/completed/failed timestamps와 `.omc/state/team/<teamName>` 상태 루트를 갖고, worker death transition을 requeued/failed/skipped로 모델링한다([omc 루프](../harnesses/omc.md#루프-엔지니어링), [src/team/runtime.ts#L92-L119](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/runtime.ts#L92-L119), [src/team/runtime.ts#L251-L263](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/runtime.ts#L251-L263)). Stop 훅의 persistent-mode는 ralph, ultragoal, autopilot, swarm, ultrawork, team 같은 active mode state를 읽고, context/user/auth/scheduled stop 예외를 먼저 통과시킨 뒤 필요하면 block decision을 낸다([omc 루프](../harnesses/omc.md#루프-엔지니어링), [persistent-mode.mjs#L949-L999](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/persistent-mode.mjs#L949-L999), [persistent-mode.mjs#L1001-L1108](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/scripts/persistent-mode.mjs#L1001-L1108)).

### gajae-code: 명시적 workflow chain과 receipt-based completion

GJC는 deep-interview, ralplan, ultragoal, team을 서로 다른 루프로 분리한다. deep-interview는 Round 0 topology gate와 weakest-dimension 질문 회전을 거쳐 threshold 이하가 될 때까지 실행을 막고, ralplan은 Planner → Architect → Critic loop를 pending approval로 멈춘다([gajae-code 루프](../harnesses/gajae-code.md#루프-엔지니어링), [deep-interview/SKILL.md#L40-L55](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/deep-interview/SKILL.md#L40-L55), [ralplan/SKILL.md#L56-L89](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/ralplan/SKILL.md#L56-L89)). ultragoal은 `.gjc/ultragoal/goals.json`과 `ledger.jsonl`로 story checkpoint를 관리하며, complete checkpoint는 architectReview/executorQa/iteration 구조와 fresh goal snapshot을 runtime이 검사한다([gajae-code 루프](../harnesses/gajae-code.md#루프-엔지니어링), [ultragoal/SKILL.md#L86-L104](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/defaults/gjc/skills/ultragoal/SKILL.md#L86-L104), [ultragoal-runtime.ts#L996-L1062](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/gjc-runtime/ultragoal-runtime.ts#L996-L1062)). Skill handoff도 caller phase가 terminal/handoff 범위일 때만 허용되어, 루프 간 전환 자체가 state guard를 통과해야 한다([gajae-code 루프](../harnesses/gajae-code.md#루프-엔지니어링), [tools/skill.ts#L90-L151](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/skill.ts#L90-L151)).

### ouroboros: interview → execute → evaluate → evolve의 세대 루프

Ouroboros는 스펙 확정 전 루프와 실행 후 루프를 모두 workflow OS 안에 둔다. Big Bang ambiguity gate는 goal/constraint/success criteria 가중 clarity가 threshold 이하일 때 Seed generation을 허용하고, interview engine은 prompt budget safety와 resume state를 유지한다([ouroboros 루프](../harnesses/ouroboros.md#루프-엔지니어링), [ambiguity.py#L34-L57](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/ambiguity.py#L34-L57), [interview.py#L162-L192](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/bigbang/interview.py#L162-L192)). 실행은 Double Diamond phase machine이고, evaluation은 mechanical → semantic → consensus 순서로 멈춤 조건을 둔다([ouroboros 루프](../harnesses/ouroboros.md#루프-엔지니어링), [double_diamond.py#L1-L16](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/execution/double_diamond.py#L1-L16), [pipeline.py#L91-L164](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/evaluation/pipeline.py#L91-L164)). 마지막으로 evaluation output이 다음 generation seed로 들어가는 evolution loop와 stagnation pattern detector가 붙어 반복 실패의 모양도 루프 상태로 다룬다([ouroboros 루프](../harnesses/ouroboros.md#루프-엔지니어링), [README.md#L232-L257](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/README.md#L232-L257), [stagnation.py#L153-L245](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/resilience/stagnation.py#L153-L245)).

## 분기 원인 분석

첫 번째 원인은 **루프가 끼어드는 위치**다. opencode는 LLM 응답 처리 loop 내부에서 stop/continue/compact를 판정하고, omo는 opencode event hook으로 idle 이후의 종료 주장을 재검사하며, lazy-codex와 omc는 host가 제공하는 Stop 계열 이벤트에서 종료를 block한다([opencode 루프](../harnesses/opencode.md#루프-엔지니어링), [omo 루프](../harnesses/omo.md#루프-엔지니어링), [lazy-codex 루프](../harnesses/lazycodex.md#루프-엔지니어링), [omc 루프](../harnesses/omc.md#루프-엔지니어링)).

두 번째 원인은 **완료 근거의 저장 위치**다. omo는 세션 내부 state와 hook condition을 주로 쓰고, lazy-codex는 `.omo` 파일과 transcript 문자열 검사를 쓴다. omc는 `.omc/state/team` task record와 worker lifecycle을 쓰며, GJC는 `.gjc/ultragoal` ledger와 completion receipt를 쓴다([omo 루프](../harnesses/omo.md#루프-엔지니어링), [lazy-codex 루프](../harnesses/lazycodex.md#루프-엔지니어링), [omc 루프](../harnesses/omc.md#루프-엔지니어링), [gajae-code 루프](../harnesses/gajae-code.md#루프-엔지니어링)). 파일·ledger로 내려갈수록 세션 중단 뒤 재개와 감사가 쉬워지지만, 루프 구현은 여러 command와 state writer로 나뉜다.

세 번째 원인은 **검증을 루프 앞에 둘지 뒤에 둘지**다. GJC와 Ouroboros는 요구사항 단계에서 ambiguity/clarity gate를 먼저 통과시킨 뒤 실행 루프로 넘어간다([gajae-code 루프](../harnesses/gajae-code.md#루프-엔지니어링), [ouroboros 루프](../harnesses/ouroboros.md#루프-엔지니어링)). opencode·omo·lazy-codex·omc는 실행 중단·완료·재시도 주변에서 루프 guard를 더 많이 다룬다([opencode 루프](../harnesses/opencode.md#루프-엔지니어링), [omc 루프](../harnesses/omc.md#루프-엔지니어링)). 이 차이는 “실행 전에 모호성을 줄이는 하네스”와 “실행 중 완료 주장을 계속 검사하는 하네스”의 분기로 이어진다.

## 비교 메모

루프 엔지니어링의 공통 질문은 “모델이 멈추려 할 때 무엇이 다시 실행을 요구하는가”다. opencode는 응답 구조와 토큰 상태, omo는 todo와 completion promise, lazy-codex는 Stop 훅과 파일 상태, omc는 Team stage와 persistent mode, GJC는 workflow phase와 receipt, Ouroboros는 ambiguity·evaluation·evolution gate가 그 답이다. 같은 자동화라도 루프의 근거가 메시지, hook, 파일, ledger, Seed로 이동하면서 하네스의 성격이 달라진다.
