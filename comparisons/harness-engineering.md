# 하네스 엔지니어링 횡단 비교

## 차원 정의

[framework.md의 하네스 엔지니어링 정의](../framework.md#1-하네스-엔지니어링-harness-engineering)는 모델 호출 바깥에서 에이전트의 행동 가능 범위와 입력 품질을 정하는 정적 기반 설비를 가리킨다. 따라서 이 에세이는 도구 표면, 권한 경계, 컨텍스트 공급, MCP·LSP·tmux 같은 환경 통합이 각 하네스에서 어떤 형태로 고정되는지를 비교한다.

## 6개 하네스의 설계 분기

### opencode: 중립 베이스의 도구·권한·컨텍스트 기본형

opencode의 하네스 설계는 파생 플러그인이 기대는 기준면이다. 내장 도구 배열은 `invalid`, `question`, `shell`, `read`, `glob`, `grep`, `edit`, `write`, `task`, `fetch`, `todo`, `search`, `skill`, `patch`와 조건부 `lsp`/`plan`을 한 registry로 묶고, `Tool.define`이 스키마 검증·invalid argument 표면·출력 truncation을 공통 처리한다([opencode 하네스](../harnesses/opencode.md#하네스-엔지니어링), [registry.ts#L219-L236](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/tool/registry.ts#L219-L236), [tool.ts#L24-L34](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/tool/tool.ts#L24-L34)). read는 라인 번호 프리픽스와 크기 상한으로 파일 세계를 모델에게 재포맷하고, edit는 9종 Replacer 체인과 파일별 세마포어로 문자열 치환 실패를 흡수한다([opencode 하네스](../harnesses/opencode.md#하네스-엔지니어링), [read.ts#L13-L19](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/tool/read.ts#L13-L19), [edit.ts#L694-L704](https://github.com/sst/opencode/blob/2c5335d84c2048902022c918b738b84c7220c561/packages/opencode/src/tool/edit.ts#L694-L704)).

### omo: opencode 표면을 확장하는 플러그인 증폭기

omo는 opencode의 기본 도구 표면 위에 `call_omo_agent`, `look_at`, `delegate_task`, `task_*`, `hashline_edit`, 팀 메시징, 백그라운드 조회·취소 도구를 더하고, registry에서 코어 도구·팀 모드·태스크 시스템·해시라인 도구를 조건부 결합한다([omo 하네스](../harnesses/omo.md#하네스-엔지니어링), [tools/index.ts#L24-L43](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/index.ts#L24-L43), [tool-registry.ts#L51-L64](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/plugin/tool-registry.ts#L51-L64)). 가장 선명한 분기는 hashline이다. read 출력 뒤에 `LINE#HASH|content` 좌표계를 붙이고 `hashline_edit`가 그 anchor를 받아 편집하므로, 모델이 오래된 문맥을 짚을 때 문자열 유사도 복구 대신 anchor 불일치로 거부한다([omo 하네스](../harnesses/omo.md#하네스-엔지니어링), [hashline-read-enhancer/hook.ts#L56-L66](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/hooks/hashline-read-enhancer/hook.ts#L56-L66), [hashline-edit/tools.ts#L21-L38](https://github.com/code-yeongyu/oh-my-openagent/blob/a4dba62b1afd4ba116b6a7f5ce2a6d4d45279a45/packages/omo-opencode/src/tools/hashline-edit/tools.ts#L21-L38)).

### lazy-codex: Codex 확장점에 맞춘 외부 프로세스 하네스

lazy-codex는 omo의 의도를 Codex의 훅·스킬·MCP 계약으로 옮기면서, in-process 플러그인 함수 대신 `hooks.json`의 이벤트별 `node .../dist/cli.js` 명령을 사용한다([lazy-codex 하네스](../harnesses/lazycodex.md#하네스-엔지니어링), [hooks.json#L3-L35](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/hooks/hooks.json#L3-L35), [ultrawork/src/codex-hook.ts#L31-L37](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ultrawork/src/codex-hook.ts#L31-L37)). 그래서 도구를 새로 정의하기보다 기존 도구 입력을 PreToolUse에서 거부하거나, `.mcp.json`으로 ast_grep·grep_app·context7·git_bash·lsp를 동봉하고, 역할 TOML을 Codex 네이티브 agent 표면에 맞춘다([lazy-codex 하네스](../harnesses/lazycodex.md#하네스-엔지니어링), [ulw-loop/src/codex-hook.ts#L86-L98](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/components/ulw-loop/src/codex-hook.ts#L86-L98), [.mcp.json#L2-L33](https://github.com/code-yeongyu/lazycodex/blob/a596185f44e451ba3deb325097089906b67f78e3/plugins/omo/.mcp.json#L2-L33)).

### omc: Claude Code 전 이벤트 훅과 tmux 워커 표면

omc는 Claude Code 플러그인 매니페스트에서 skills, MCP, commands를 한 번에 노출하고, UserPromptSubmit·SessionStart·PreToolUse·PermissionRequest·PostToolUse·SubagentStart/Stop·PreCompact·Stop·SessionEnd까지 커맨드 훅을 배치한다([omc 하네스](../harnesses/omc.md#하네스-엔지니어링), [.claude-plugin/plugin.json#L1-L62](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/.claude-plugin/plugin.json#L1-L62), [hooks/hooks.json#L4-L210](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/hooks/hooks.json#L4-L210)). 또 `claude | codex | gemini | cursor | grok` CLI worker 타입과 binary path trust check, CLI별 approval/sandbox flag를 Team 런타임이 조립하므로, 세션 안 Task 표면과 바깥 tmux worker 표면이 같은 하네스 안에 공존한다([omc 하네스](../harnesses/omc.md#하네스-엔지니어링), [src/team/model-contract.ts#L8-L20](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/model-contract.ts#L8-L20), [src/team/model-contract.ts#L172-L219](https://github.com/Yeachan-Heo/oh-my-claudecode/blob/deee3a446dadc9bfea31cdc8b19b00b16718082e/src/team/model-contract.ts#L172-L219)).

### gajae-code: 외부 runner가 직접 소유하는 세션 도구 표면

GJC는 다른 런타임에 플러그인으로 들어가지 않고 `gjc` 프로세스가 도구 map을 직접 만든다. public callable map에는 read, bash, edit, AST/LSP, browser, task/subagent, skill, goal, web search 등이 포함되고, ToolSession은 cwd·workspace tree·skill state·goal runtime·model settings·subagent context를 한 구조체로 묶는다([gajae-code 하네스](../harnesses/gajae-code.md#하네스-엔지니어링), [tools/index.ts#L313-L348](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/index.ts#L313-L348), [tools/index.ts#L119-L240](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/index.ts#L119-L240)). read는 URL·문서·notebook·archive·SQLite·code summary를 포괄하는 context 공급 장치이고, edit는 replace/patch/hashline/vim/apply_patch 문법과 LSP writethrough를 한 도구 안에 결합한다([gajae-code 하네스](../harnesses/gajae-code.md#하네스-엔지니어링), [read.ts#L1-L90](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/tools/read.ts#L1-L90), [edit/index.ts#L1-L57](https://github.com/Yeachan-Heo/gajae-code/blob/0464f0f92a278ed34059db48d9951cbf41c1ce78/packages/coding-agent/src/edit/index.ts#L1-L57)).

### ouroboros: 스펙을 context substrate로 승격한 Agent OS 표면

Ouroboros의 하네스 표면은 단순 tool registry보다 넓다. `ooo` skill 라우팅과 Typer CLI command group이 command space를 만들고, Claude plugin wrapper는 skills와 MCP server를 사용자 런타임에 싣는다([ouroboros 하네스](../harnesses/ouroboros.md#하네스-엔지니어링), [AGENTS.md#L10-L38](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/AGENTS.md#L10-L38), [cli/main.py#L74-L112](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/cli/main.py#L74-L112), [.claude-plugin/plugin.json#L1-L24](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/.claude-plugin/plugin.json#L1-L24)). 하지만 더 중요한 하네스 장치는 Seed, EventStore, Workflow IR다. Seed는 goal·constraints·acceptance criteria를 immutable specification으로 묶어 evaluation의 ground truth가 되고, Workflow IR는 agent/plugin/verifier work를 dispatch하기 전 typed node와 evidence contract로 감싼다([ouroboros 하네스](../harnesses/ouroboros.md#하네스-엔지니어링), [seed.py#L182-L199](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/core/seed.py#L182-L199), [workflow_ir.py#L248-L309](https://github.com/Q00/ouroboros/blob/b032f3a96fa2a5388affd55a2042a5a439fbe5f3/src/ouroboros/orchestrator/workflow_ir.py#L248-L309)).

## 분기 원인 분석

첫 번째 원인은 **host runtime의 확장 깊이**다. opencode는 플러그인 훅과 공개 SDK 경계를 제공하므로 omo는 tool registry, read 후처리, MCP 설정 재작성까지 in-process로 증폭할 수 있다([opencode 아키텍처](../harnesses/opencode.md#아키텍처), [omo 하네스](../harnesses/omo.md#하네스-엔지니어링)). 반대로 lazy-codex는 Codex가 허용하는 이벤트 훅·MCP·agent TOML 안에서 움직여야 하므로, 도구 정의보다 외부 프로세스 훅과 권한 거부가 중심이 된다([lazy-codex 하네스](../harnesses/lazycodex.md#하네스-엔지니어링)).

두 번째 원인은 **권위의 위치**다. 플러그인형 하네스는 host가 세션과 도구 호출 권위를 가진다. 그래서 opencode·omo·lazy-codex·omc의 하네스 장치는 host가 부르는 훅, 매니페스트, MCP, agent file로 표현된다([opencode 아키텍처](../harnesses/opencode.md#아키텍처), [omc 하네스](../harnesses/omc.md#하네스-엔지니어링)). GJC와 Ouroboros는 외부 runner·OS로 자신을 놓기 때문에, `.gjc`/Seed/Ledger/Workflow IR처럼 host 바깥의 상태 계약이 하네스 표면의 중심으로 이동한다([gajae-code 아키텍처](../harnesses/gajae-code.md#아키텍처), [ouroboros 하네스](../harnesses/ouroboros.md#하네스-엔지니어링)).

세 번째 원인은 **편집·검증 위험을 어디서 줄이는가**다. opencode는 fuzzy Replacer 체인으로 edit 실패를 복구하고, omo와 GJC는 hashline anchor 무결성으로 stale write를 거부하며, Ouroboros는 Seed와 Workflow IR의 evidence contract로 실행 전후 맥락을 고정한다([opencode 하네스](../harnesses/opencode.md#하네스-엔지니어링), [omo 하네스](../harnesses/omo.md#하네스-엔지니어링), [gajae-code 하네스](../harnesses/gajae-code.md#하네스-엔지니어링), [ouroboros 하네스](../harnesses/ouroboros.md#하네스-엔지니어링)). 같은 문제를 문자열 복구, 좌표 검증, 스펙 계약이라는 서로 다른 하네스 층에서 다룬 셈이다.

## 비교 메모

하네스 엔지니어링의 축은 “도구가 많다”가 아니라 “모델이 접촉하는 세계를 누가 어떤 계약으로 재작성하는가”다. opencode는 중립 substrate, omo는 그 substrate의 확장, lazy-codex는 제한된 host API에서의 외부 훅 재구성, omc는 Claude Code 이벤트·tmux worker mesh, GJC는 외부 runner 세션 도구, Ouroboros는 Seed/Workflow IR 기반 context substrate로 갈라진다. 이 분기는 이후 루프·프롬프트·아키텍처·모델 튜닝의 차이도 함께 만든다.
