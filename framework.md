# 비교 프레임워크 — v0 (가설)

> **지위: v0 (가설).** 이 문서의 차원 정의는 분석 대상 코드에 본격적으로 접촉하기 전에 작성되었습니다. 심층 분석 2편 완료 시점(C1 회고)과 6편 완료 시점(C2 동결)에 실제 분석에서 드러난 경계 사례를 반영해 수정되며, C2에서 **freeze** 선언이 이루어진 뒤에야 확정 정의가 됩니다. freeze 전까지 모든 정의는 반증 가능한 가설로 취급하세요.

이 문서는 `harnesses/`의 심층 분석 6편과 `comparisons/`의 횡단 에세이 5편이 공유하는 **단일 분류 기준**입니다. 모든 문서는 "이 코드가 어느 차원에 속하는가"를 판단할 때 이 문서의 정의와 판별 질문을 참조하고, 정의와 충돌하는 사례를 발견하면 문서를 고치는 게 아니라 C1/C2 회고 안건으로 올립니다.

## 5개 비교 차원의 조작적 정의

### 1. 하네스 엔지니어링 (Harness Engineering)

**정의:** 모델 호출 *바깥*에서 에이전트의 행동 가능 범위와 입력 품질을 결정하는 정적 기반 설비. 도구 표면(어떤 도구가 어떤 파라미터로 노출되는가), 권한·샌드박스 경계, 컨텍스트 공급 장치(파일 읽기 형식, repo 맵, 메모리/지식 주입), 환경 통합(LSP, AST, tmux, MCP)이 여기 속한다.

**판별 질문:**
- 이 코드는 모델이 *무엇을 할 수 있는지*(도구·권한)를 바꾸는가? → 하네스
- 이 코드는 모델에게 *세계를 어떻게 보여줄지*(컨텍스트 형식·공급)를 바꾸는가? → 하네스
- 단일 턴 안에서 동작하고, 반복 횟수·종료 여부와 무관한가? → 하네스 쪽 신호

### 2. 루프 엔지니어링 (Loop Engineering)

**정의:** 에이전트 실행의 *시간 축*을 제어하는 동적 설비. 반복 제어(언제 한 번 더 도는가), 종료 조건(무엇이 "끝"인가), 재시도·복구 전략, 검증 사이클(완료 주장을 어떻게 검사하고 반려하는가), 체크포인트·이어가기, 서브에이전트 오케스트레이션의 수명 주기가 여기 속한다.

**판별 질문:**
- 이 코드는 *한 번 더 실행할지/멈출지*를 결정하는가? → 루프
- 이 코드는 완료 주장을 *검증하고 반려*하는가? → 루프
- 이 코드를 제거하면 에이전트가 같은 일을 하되 더 일찍/늦게 멈추는가? → 루프

**하네스 vs 루프 경계 (가장 흔한 혼동):** 도구 표면·권한·컨텍스트 공급 = **하네스**, 반복 제어·종료 조건·재시도와 검증 사이클 = **루프**. 예컨대 "edit 도구에 라인 해시 검증을 넣는 것"은 도구 표면의 성질이므로 하네스이고, "edit 실패 시 재읽기 후 재시도를 강제하는 것"은 복구 전략이므로 루프다. 서브에이전트의 경우: 어떤 에이전트 종류와 도구를 노출하는가는 하네스, 스폰·await·재개·취소의 수명 주기 규칙은 루프.

### 3. 프롬프트 엔지니어링 (Prompt Engineering)

**정의:** 모델에게 주입되는 *자연어 지시문의 내용과 구조*. 시스템 프롬프트의 정체성·규칙·금지 패턴, 도구 설명문, 스킬/커맨드 프롬프트, 역할 에이전트의 페르소나, 키워드 트리거가 주입하는 지시문, 출력 형식 강제가 여기 속한다.

**판별 질문:**
- 이 산출물은 결국 모델 컨텍스트에 *텍스트로* 들어가는 지시문인가? → 프롬프트
- 같은 텍스트를 다른 하네스에 옮겨 붙여도 효과가 유지되는가? → 프롬프트 쪽 신호 (하네스 의존 장치라면 하네스/아키텍처)
- 코드 변경 없이 이 파일만 수정해 행동이 바뀌는가? → 프롬프트

### 4. 아키텍처 (Architecture)

**정의:** 하네스를 *구성하는 방식* 자체. 배포 형태(플러그인 vs 독립 실행형 vs CLI 래퍼), 모듈 경계와 의존 방향, 상태 저장 위치와 형식, 확장점 설계(훅, 플러그인 API), 베이스 플랫폼과의 결합도, 멀티 패키지/멀티 배포판 전략이 여기 속한다.

**판별 질문:**
- 이 결정은 개별 기능이 아니라 *기능들이 놓이는 골격*에 관한 것인가? → 아키텍처
- 이 결정을 바꾸려면 여러 모듈을 동시에 옮겨야 하는가? → 아키텍처
- "어디에 무엇이 사는가"(상태, 설정, 확장점)에 대한 답인가? → 아키텍처

### 5. 모델별 튜닝 (Per-Model Tuning)

**정의:** *특정 모델·모델 계열의 성질*에 맞춘 분기와 최적화. 모델 라우팅(작업 종류→모델 선택), 폴백 체인, 모델별 프롬프트 변형, 토큰 예산·컨텍스트 윈도 전략, 모델별 도구 호출 형식 대응, 비용·쿼터 규율이 여기 속한다.

**판별 질문:**
- 이 코드/프롬프트에 모델 이름·계열·가격에 따른 분기가 있는가? → 모델별 튜닝
- 모델이 하나뿐이라면 이 코드는 무의미해지는가? → 모델별 튜닝
- "어떤 일에 어떤 두뇌를 쓰는가"에 대한 답인가? → 모델별 튜닝

## 인용 규칙

1. **모든 설계 주장에는 소스 근거가 필수다.** 근거는 파일 경로, 프롬프트 원문 인용, 코드 인용 중 1개 이상이며, 링크는 SHA 고정 permalink로 작성한다.
2. **permalink 형식:** `https://github.com/<owner>/<repo>/blob/<sha>/<path>#L<n>` — 단일 라인. 라인 범위는 `#L<n>-L<m>`. `<sha>`는 반드시 부록 "분석 기준 커밋" 표의 고정 SHA여야 한다.
3. **부재 증명:** "이 하네스에는 X 설계가 없다"는 주장도 근거가 필요하다. 탐색한 경로·패턴(예: "`src/` 전체에서 `model` 라우팅 패턴 검색, 설정 스키마 확인")을 함께 기록한다. 매트릭스와 본문에서 빈 셀·무근거 부재 주장은 금지.
4. **클론 채록일 병기:** 각 분석 문서 머리에 분석 기준 커밋 SHA와 클론 채록일을 함께 적는다 (고정 헤더 패턴: `분석 기준 커밋:`).
5. **중대 upstream 변경 주석:** 분석 후 원격 저장소에 개명·아키텍처 개편 등 중대 변경을 인지하면, 재분석 없이 문서 머리에 1줄 주석으로만 기록한다.
6. **추정 표시:** 인용으로 뒷받침되지 않는 해석·추정은 "추정"임을 문장에 명시하거나 삭제한다.

### 자가 체크와 표본 검토 (AC3b)

- 각 문서 작성자는 문단 탈고 시 "이 문단의 설계 주장에 permalink 인용이 붙어 있는가"를 확인하고, 문서 말미에 "전 주장 인용 자가 점검 완료"를 기록한다.
- **표본 검토 주체:** 표본 3건의 주장-인용 대응 검토는 문서 작성자가 아닌 검토 레인(ultragoal 품질 게이트의 architect/executor QA 레인, 또는 그에 준하는 독립 검토자)이 수행한다. 작성자 자신의 표본 검토는 무효다.

## 기준선 스코프 규칙

`harnesses/opencode.md`(기준선)는 백과사전이 아니다. **파생·플러그인 하네스(omo, lazy-codex 등) 비교에 필요한 설계 표면으로 한정**한다: 플러그인/훅 확장점, 도구 표면의 기본형, 에이전트 루프의 기본형, 프롬프트 기본 골격, 모델 추상화. opencode 자체의 모든 기능을 망라하려는 시도는 스코프 위반이다.

## 매트릭스 셀 작성 규칙

README의 6×5 매트릭스 셀은 **가치중립 설계 접근 명사구 + 심층 문서 앵커 링크**만 담는다. 등급·점수·별점·순위·추천 표현("우수", "최고", "미흡" 등 가치 판단 형용사 포함)은 금지. 셀 키워드는 해당 하네스가 그 차원에서 택한 설계 접근을 기술하는 명사구여야 한다 (예: "카테고리 기반 모델 라우팅", "라인 해시 검증 편집").

## freeze 해제 절차

C2에서 freeze가 선언된 뒤 정의를 변경하려면: (1) 변경 사유와 충돌 사례를 이 문서의 개정 이력에 기록하고, (2) 변경된 정의로 `harnesses/` 6편과 `comparisons/` 5편 전체를 소급 점검해 영향받는 문단을 함께 수정하며, (3) 두 작업을 하나의 커밋으로 묶는다. 소급 점검 없는 동결 후 정의 변경 커밋은 무효다.
