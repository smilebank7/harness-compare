# 제작자 관점과 비교 사고법

## 학습 목표

이 장의 목표는 하네스를 소비자 관점의 추천 대상으로 보지 않고, 제작자 관점의 설계 결정 묶음으로 읽는 사고법을 익히는 것입니다. 독자는 “어느 도구가 더 좋은가?” 대신 “어떤 문제를 어떤 경계와 루프로 풀었는가?”를 질문할 수 있어야 합니다.

## 요약

`harness-compare`는 하네스 선택 가이드가 아닙니다. 여섯 하네스가 서로 다른 substrate, loop, prompt contract, architecture, model routing을 어떻게 구성했는지 비교합니다. 좋은 비교는 칭찬이나 순위가 아니라 **가치중립 명사구 + 근거 링크 + tradeoff 설명**으로 구성됩니다.

## 핵심 개념

- **제작자 관점**: 사용성 평가보다 설계자가 선택한 경계, 상태, 루프, 모델 분기, 검증 방식을 봅니다.
- **가치중립 명사구**: “강력한 도구”가 아니라 “기본 도구·권한 substrate”처럼 접근을 이름 붙입니다.
- **차원별 분해**: 하나의 하네스를 5개 차원으로 나누어 읽으면 과장된 총평을 피할 수 있습니다.
- **근거 우선 비교**: 비교 문장은 기존 분석 문서나 framework 링크로 되돌아갈 수 있어야 합니다.

## 설계 패턴

### Dimension-first decomposition

복잡한 하네스를 한 문장으로 평가하려 하면 “좋다/나쁘다”로 흐릅니다. 먼저 [framework](../../framework.md)의 5개 차원으로 분해합니다. 같은 하네스라도 도구 표면은 넓고, 루프 종료 조건은 약할 수 있습니다. 차원별로 보면 설계 tradeoff가 보입니다.

### Neutral label matrix

README의 요약 매트릭스는 각 셀을 가치중립 설계 접근 명사구로 제한합니다. 이는 독자가 판단을 외주화하지 않고, 링크를 따라가 직접 근거를 보게 만드는 장치입니다.

## 다이어그램

```mermaid
  A[총평 욕구] --> B[5개 차원으로 분해]
  B --> C[가치중립 명사구]
  C --> D[근거 링크 확인]
  D --> E[tradeoff 서술]
  E --> F[내 설계 결정]
```

캡션: 제작자 관점 비교는 총평을 차원별 판단, 근거 확인, tradeoff 서술로 바꿉니다.

텍스트 설명: 어떤 하네스가 더 낫다고 말하기 전에 5개 차원으로 분해하고, 각 차원의 명사구와 근거 문서를 확인한 뒤 내 설계 결정으로 옮깁니다.

## 핵심 질문

- 이 문장은 평가인가, 설계 접근의 이름인가?
- 내가 비교하는 항목은 같은 차원에 속하는가?
- 이 판단은 어떤 기존 분석 문서나 source permalink로 돌아갈 수 있는가?
- 내 설계안은 어떤 tradeoff를 받아들이고 어떤 tradeoff를 피하는가?
